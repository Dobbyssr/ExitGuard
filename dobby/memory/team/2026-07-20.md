### 확정 소스 2종 (도비가 근거 URL·실물 직접 검증)
- **L1 법령 — 국가법령정보센터 OpenAPI (open.law.go.kr, 1순위/확실)**: "영리 목적 이용 포함 자유 활용 보장" 명시(도비 WebFetch 확인). 무료·출처표시 의무. 근기법 §36·§27·§26 조문 100% 커버. **인증키(OC) 발급 = 회원가입→활용신청→담당자 심의 1~2일**, 이건 대표님 개인계정 인증이라 도비 대행 불가. 단서: "개별 API별 제한 가능" — 판례(법원) target 상업이용만 개별 미확정(MVP엔 불필요).
- **L2 판정례 — 중노위 주요판정사례 (data.go.kr 15143186, 2순위)**: "이용허락범위 제한 없음 = 상업 이용 가능" 명시(도비 확인). 로그인 없이 CSV 즉시 다운로드 성공(오픈API XML/JSON도 실재하나 활용신청 승인 필요). 399건.

### L2 CSV 실측 (도비 PowerShell Import-Csv 재검증 — 론 4개 주장 전부 통과)
- 헤더: `구분·순번·자료구분·제목·위원회명·작성일자·조회수` / 399행 / 전부 "중앙"(지노위 없음).
- 자료구분 분포 검증 일치: 해고계열(징계87+통상43+경영상12)=**142건(36%)**, 부당노동행위 74 등.
- **§36 금품청산 = 0건 확정**(금품청산·퇴직금·14일·36조 매칭 전부 0) → 이 소스는 §36 미커버. **구조적 공백(§36=임금체불 형사·진정 영역, 중노위 부당해고 판정 대상 아님) → 문제 아님, §36은 법령 API로 커버.**
- **§27 서면통지 부당해고 사례 15건 실존.** 데모 직결 예시: 순번 388(2025, 서면 미교부→부당해고, 근기법 §27 직접 인용), 순번 51(2015, **휴대폰 문자 통보→서면통지 위반 부당** = 구두통보 시나리오에 최적).

### 데이터 성격·한계 (트레이드오프)
- **즉시 가능**: 출처 배지 + "공개 판정례에서 이 요소 누락 시 부당해고 판정 사례가 있습니다"(직역법 허용 프레이밍) 사례 인용 한 줄.
- **부족**: 사건번호 73% 결여(제목 텍스트 정규식 의존), 판정요지=한 줄 요약뿐(요구요소 세분 매핑엔 근거 얕음), 인용조문 구조화 필드 없음. → "체크항목별 요구요소 매핑"은 제목 텍스트 파싱 로직 별도 필요.
- 인코딩 함정(구현 시 인계): CSV 실제 바이트 = **CP949(EUC-KR)**(헤더는 UTF-8 표기, 오독 주의) + 제목에 콤마 포함 → **RFC4180 quote-aware 파서 필수**(naive split 금지).

### ★TOP3 (도비 검증: 전부 타당 — 세로절단 종반부·해자 계약이 얇음)
- **T1 [보강] compare 엔진 LLM/결정론 경계 계약 부재** — 코어 data-model §6-3에 "LLM=인용·링크만" 제약은 있으나, **무엇을 LLM이(비정형 텍스트→신호추출·근거생성)/무엇이 결정론(요소→판례매핑·5행 shape)인지 역할분담 계약이 없음.** PRODUCT §3-2가 해자로 못박은 "AI 사실추출·매핑·근거생성"의 방어선이 구현자 재량으로 흔들림 = "GPT 래퍼 아님" 심사 1순위 질문 직결. → 코어 §6에 LLM 역할·입출력·직역법 제약 명시.
- **T2 [보강] 증적 자동 봉인 트리거 계약 부재** — api-spec에 수동 `POST /cases/{id}/evidence` 1건만(도비 확인). CM-12/PRODUCT §3-1의 "증적 **자동** 축적"이 미계약 — 각 처리(상신확인·회수확인·대조결과)가 언제 자동 Evidence append+prev_hash 체인 잇는지 규약 없음 + compare 결과 영속화·증적 편입 미정. → 코어 data-model §3-4/api-spec §2-5에 이벤트→자동봉인 규칙.
- **T3 [경량신설] 게이트 통과→퇴사 승인 확정 플로우 부재** — `Case.status`(in_progress/review_waiting/completed) 필드는 있으나 **전이 트리거·엔드포인트 없음**(도비 확인). `defensible=true`는 파생 boolean뿐. PRODUCT "세 레일 충족돼야 승인 강제"가 미구현. → Case 상태전이 규칙 + `POST /cases/{id}/approve`(경량).

### (B) 데모·심사 유리
- **B1 [보강] 방어리포트 뷰 스키마 부재** — CM-13 export가 "완료율·리스크·이력 포함" 한 줄뿐. DefenseReport 스키마(KPI·3레일 배지·판례인용·증적체인 요약·**boundary 고지**) 없음. 머니샷+외부제출 문서라 boundary 누락 시 직역법 위험.
- **B2 [보강] 프로파일·템플릿 시드 통합 계약** — 접수 시 items 자동생성 근원(김민준 profile→rail_map→3 RailTemplate→TemplateItem)이 레일별로 흩어짐. CM-22(빈종이 금지=해자)의 실체.
- **B3 [신설] CM-33 머니샷(두 케이스 게이트 대조)** — 두번째 시드 케이스+대조 명세 없음(엔드포인트는 `GET /cases` 재사용).
