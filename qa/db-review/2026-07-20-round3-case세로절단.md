## A. round2 예약 게이트 착지 확인 (도비 지시 필수 항목)

### A-1. seq 채번 `SELECT … FOR UPDATE` 직렬화 — **✅ 완전 착지 (핵심 확인)**

**대상:** `evidence/repository.py:21-36` `_lock_case`+`_get_last`, `append():56-69`

**동작 분석(왜 seq 단조가 보장되나):**
`append()`는 `_lock_case`(해당 Case 행 `SELECT id … FOR UPDATE`) → `_get_last`(seq desc limit 1) → 조립 → insert → flush 순서다. 동시 append 2건(T1·T2, PG 기본 READ COMMITTED)에서:
```
T1: _lock_case → cases[id] 행잠금 획득       T2: _lock_case → 같은 행 잠금 시도 → 블록(대기)
T1: _get_last → max seq=N                     T2: (블록 중)
T1: insert seq=N+1, flush
T1: (요청 종료) COMMIT → 행잠금 해제
                                              T2: 잠금 획득 → _get_last(새 스냅샷) → seq=N+1 ← T1 커밋 반영
                                              T2: insert seq=N+2
```
`FOR UPDATE` 행잠금은 **트랜잭션 커밋까지 유지**되고, READ COMMITTED는 매 문장이 새 스냅샷을 뜨므로 블록에서 풀린 T2의 `_get_last`는 T1이 커밋한 seq를 본다. **→ seq 단조·prev_hash 단일체인 보장.** Case당 잠금이라 케이스 간 병렬성 손해 없음(§3-4-1(2) 정확 구현).

**부수 확인 — 서비스 경로가 이 잠금을 더 강화한다:** `submit_item`·`review_item`·`approve_case`는 append **이전에** `_recompute_case_status`(또는 `approve_case`의 `case.status=completed`)로 이미 cases 행을 UPDATE·flush한다. 즉 append의 FOR UPDATE 전에 동일 행에 배타락이 걸려 있어 동시성 직렬화가 append 지점보다 앞서 성립한다. 자기 트랜잭션이 이미 잠근 행을 FOR UPDATE로 재잠금하는 것은 무해. **정합.**

**실물:** `test_concurrent_append_serializes_seq_via_row_lock` 통과 — asyncio.gather 2건이 seq [1,2], `entries[1].prev_hash == entries[0].integrity_hash`(체인 미분기) 확인.

### A-2. 목록 게이트 N+1 회피 — **✅ 완전 착지**

**대상:** `case/repository.py:70-92` `get_item_counts` + `case/service.py:249-264` `list_cases`

`list_cases`는 케이스 목록을 1쿼리로 뽑고(`list_cases`), 그 case_id 전체를 `get_item_counts`에 **한 번에** 넘겨 `WHERE case_id IN (...) GROUP BY case_id, rail, status, blocking` **단일 GROUP BY**로 집계한다. 케이스별 items 반복 로드(N+1)·ORM lazy-load(`case.items`) 순회 **없음**. round1 🟡-1·round2 C-3·§0 (o) ③ 권고 정확 반영. **정합.**

### A-3. `Case.status` 항목전이 동일 트랜잭션 재계산 — **✅ 착지 (round1 🟡-4)**

`submit_item`·`review_item`이 항목 전이 flush 직후 `_recompute_case_status`(`recompute_status` §3-1-1)를 **같은 세션**에서 호출 → desync 없음. `completed` 종단상태 보존(`recompute_status:127-128`)도 §3-1-1 일치. `updated_at` 명시 설정(async onupdate MissingGreenlet 회피)은 정당한 구현 처리.

## B. 이번 라운드 신규 쿼리·로직 6렌즈 감수

### B-1. GROUP BY 집계쿼리가 §5 게이트를 정확·완전 재현하나 — **✅ 완전 일치**

`get_item_counts`의 (case_id,rail,status,blocking)×count 집계행을 `compute_gate_from_counts`(service.py:81-114)가 소비한다. §5 계산식과 라인별 대조:

| §5 계산식 | 구현 | 일치 |
|---|---|---|
| `applicable = count(status != na)` | `if c.status != not_applicable: rail_applicable += count` | ✅ |
| `approved = count(status == approved)` | `if c.status == approved: rail_approved += count` | ✅ |
| `rail_completion = round(100*approved/max(1,applicable))` | 동일 | ✅ |
| `overall = round(100*Σapproved/max(1,Σapplicable))` | 동일 | ✅ |
| `risk_count = count(blocking AND status ∉ {approved,na})` | `if c.blocking and c.status not in (approved, not_applicable): risk_count += count` | ✅ |
| `defensible = risk_count==0` | 동일 | ✅ |

`rail`·`blocking`을 그룹키에 포함했기에 집계행만으로 §5 전 항이 복원된다. **의도적 비정규화(집계행 = 실물 Item 대체 입력)**가 상세(`compute_gate`, count=1×N)와 목록(집계행)을 **동일 순수함수**로 통과시키는 설계는 정합성·재사용 모두 모범. 실물: `test_gate_compute.py` 5건 통과.

### B-2. 🔵 GROUP BY의 인덱스 커버리지 — **부분커버로 충분, 추가 인덱스 불필요 [Post-MVP 관찰]**

`WHERE case_id IN (...) GROUP BY case_id, rail, status, blocking`. 기존 `ix_items_case_id_status`(case_id, status)는 **WHERE의 case_id 필터를 리딩 컬럼으로 커버**한다(인덱스로 대상 행 특정). 단 GROUP BY의 `rail`·`blocking`은 인덱스에 없어 힙 페치가 필요 → **index-only가 아니라 index scan + hash aggregate**.

- **판정: 추가 인덱스 불필요.** 커버링 인덱스 `(case_id, status, rail, blocking)`를 만들면 index-only가 되나, **case당 Item 십수 개** 카디널리티에선 힙 페치 비용이 무의미하고 쓰기비용·중복만 늘린다(YAGNI). 데모·MVP 규모에서 부분커버로 충분.
- **[Post-MVP]** 케이스 수천·item 수만 규모로 커지고 목록 집계가 핫패스가 되면 그때 커버링 인덱스 재검토. 지금은 아님.

### B-3. 🔵 `_lock_case`가 존재하지 않는 case_id에선 잠금이 안 걸린다 — **실사용상 안전, 방어적 관찰만**

**근거:** `SELECT Case.id WHERE id = <없는 id> FOR UPDATE`는 **0행 반환 → 잠글 행이 없음 → 락 no-op**. 이론상 존재하지 않는 case_id로 동시 append가 오면 직렬화가 안 된다.

**판정: 무결성 붕괴 아님 — 안전.**
- 실제 봉인은 항상 **먼저 커밋된 Case** 위에서 일어난다(`submit/review/approve`는 `get_case`로 존재 확인 후 진행, `create_case`는 append 없음). 존재하지 않는 case로 오는 append 경로가 **없다.**
- 설령 온다 해도 `evidence.case_id → cases.id` FK(NO ACTION, round2 A-3 확인)가 **INSERT 자체를 FK 위반으로 거부** → orphan 증적이 물리적으로 불가능. 락 부재가 무결성으로 새지 않는다.
- **권고 [Post-MVP/선택]:** 방어적으로 명확히 하려면 `append` 진입 시 case 존재를 assert하거나, `_lock_case`가 0행이면 `NotFoundError`를 던지는 정도. **지금은 불필요**(방어 이득 대비 코드 추가 과함, 상위 서비스가 이미 존재 확인). 심각도 🔵.

### B-4. 해시체인 알고리즘 정합 — **✅ §3-4-1 계약 일치**

- `canonical_json`: `sort_keys=True, separators=(",",":")` → 키정렬·공백정규화(재현성) 계약 충족. `ensure_ascii=False`는 한글 payload를 UTF-8 그대로 해시(아래 인코딩과 일관) — 재현성에 무해(해시 입력이 결정론적이면 됨).
- `compute_integrity_hash`: `SHA-256(canonical_json(payload) + (prev_hash or "")).encode("utf-8")` → 계약식 `SHA-256(canonical_json(payload)+prev_hash)` 정확 일치. **payload+직전해시 함께 해시 → 순서 위변조 탐지** 성립.
- `build_sealed_evidence`: `seq=(last.seq+1 if last else 1)`, `prev_hash=last.integrity_hash if last else None`(체인 시작 None) — §3-4-1 계약식 정확 일치. **순수함수(DB 무접근)**라 락 하에 조회한 `last`를 주입받는 구조 → 재현성 단위테스트 가능(`test_evidence_hashing.py` 통과).
- **round1 🟡-6 재확인:** 봉인 시점 canonical_json 문자열을 해시 입력으로 직접 만들고 JSONB 저장값에서 재직렬화하지 않는다 → 저장이 JSONB든 해시 재현성 무관. **정확히 준수.**

### B-5. 봉인 append의 트랜잭션 원자성 — **✅ 계약 "누락없는 append" 준수**

`append`는 `flush()`까지만(commit 없음, backend §4·§5). 서비스가 mutation(Approval·Item·Case.status 변경)과 봉인 append를 **동일 `get_db` 세션**에서 수행하고 커밋은 요청 종료 시 `get_db`가 1회. **→ mutation+봉인이 단일 트랜잭션 = 원자적.** 봉인이 IntegrityError(seq UNIQUE 충돌 등)로 실패하면 mutation까지 통째로 롤백 → **부분 상태·이력 누락 불가**. §3-4-1 "누락 없는 append"·round1 🔴-1(3) 준수.

### B-6. 시드 멱등성·재현 정합 — **✅ 멱등·원자·정합**

- **멱등성:** 진입 시 `_ADMIN_EMAIL` 존재 조회 → 있으면 즉시 return. 재실행 시 중복 생성 없음. **✅**
- **원자성:** 전 작업이 flush로만 쌓이고 `commit()`은 **맨 끝 1회**(seed.py:224). 중간 크래시 시 세션 컨텍스트 종료로 롤백 → 부분 시드 없음. 다음 실행은 admin 부재 → 완전 재시드. **부분상태 불가.**
- **재현 정합(§5 산수 직접 재검증):** L-09=not_applicable(권고사직 유형규칙, apply_profile 자동) → applicable = {L-01,L-02,L-04,L-06,L-08}=5, approved={L-01,L-02}=2 → labor `round(100*2/5)=40%`. ✅. L-04=submitted → `recompute_status` `has_submitted=True` → **review_waiting**. ✅. 브리프 기대치(review_waiting·labor 40%) 정확 재현.
- **증적 0 재현 정합(관찰):** 시드는 서비스(submit/review)를 **우회**해 `item.status`·`Approval`을 직접 세팅하므로 **증적 append가 발생하지 않는다 → Evidence 0건**. 이는 브리프 기대치(증적0)와 일치하는 **의도된 캘리브레이션**이다. 다만 이 케이스로 방어 리포트(§10)를 즉시 뽑으면 `evidence_chain`이 비어 있다 — DB 무결성 문제가 아니라 **데모 시작 상태의 성격**이므로 기능 QA(무디) 관점에서만 인지하면 된다.

## C. 동시성 테스트 충분성 판정 (브리프 핵심 포인트 5)

**대상:** `tests/integration/test_evidence_concurrency.py`

- **레이스 재현성 — ✅ 실제로 연다.** `asyncio.gather(_append("a"), _append("b"))` 각기 독립 세션(NullPool → 독립 커넥션 = 독립 트랜잭션). 첫 `await db.execute(_lock_case)`에서 이벤트루프가 양보하므로 **두 코루틴이 모두 FOR UPDATE를 발행한 뒤** 하나가 DB 레벨에서 블록된다 → round1이 지적한 "동일 max(seq) 동시 read" 창이 **신뢰성 있게 열린다**(스케줄러 우연에 기대지 않음).
- **회귀 검출력 — ✅ 있다.** FOR UPDATE를 제거하면 두 append가 `last=None`을 함께 읽어 둘 다 seq=1 → 두 번째 커밋이 `UNIQUE(case_id, seq)` 위반 → `asyncio.gather`가 예외 전파 → **테스트 실패(에러)**. 즉 이 테스트는 직렬화가 깨지면 반드시 빨간불이 된다. 체인 검증(`prev_hash` 연결)까지 assert해 "체인 분기"도 직접 잡는다.
- **최소성·판정:** 2-way 동시성은 직렬화 증명의 **최소 유의미 경합**이며, `test_sequential_append_is_monotonic`(순차 3건 1·2·3)이 하한을 별도 보장한다. **MVP엔 충분.**
- **[Post-MVP/선택]** 신뢰도를 더 원하면 N=5~10 스트레스 변형(gather N건 → seq가 빠짐없이 1..N) 1건 추가 정도. **지금 필수 아님**(2-way가 이미 락 유무를 가른다).
