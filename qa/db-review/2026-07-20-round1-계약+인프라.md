## 🔴 무결성/정합 — [지금 필요]

### 🔴-1. 증적 체인 `seq` 채번 Read-Modify-Write 레이스 — 체인 분기·중복 위험
**대상:** 코어 §3-4-1 append·해시 체인 규약 / `backend/app/db.py` `get_db()`

**근거(어디서 깨지나):**
계약식은 `seq = (직전 Evidence.seq of case) + 1`, `prev_hash = 직전.integrity_hash`. 이는 전형적 Read-Modify-Write다.
`get_db()`는 요청 1개=트랜잭션 1개(PostgreSQL 기본 격리 **READ COMMITTED**)로 연다. 같은 `case_id`에 처리 이벤트 2건이 거의 동시에 도착하면:

```
T1: SELECT max(seq) WHERE case_id=42  -> 7      T2: SELECT max(seq) WHERE case_id=42 -> 7
T1: INSERT seq=8, prev_hash=H7                  T2: INSERT seq=8, prev_hash=H7
T1: COMMIT (성공)                               T2: COMMIT (성공)  ← 둘 다 seq=8, 둘 다 prev=H7
```

결과: **seq 중복 + 체인 분기**(두 레코드가 같은 prev_hash를 가리켜 단일 체인이 아니게 됨). 이건 "봉인된 증거의 위변조 탐지 무결성"이라는 제품 핵심 가치를 정면으로 깬다. `head_hash` 앵커(§10)가 두 개로 갈라지면 방어 리포트의 체인 검증이 무의미해진다.

**MVP 현실감각:** 데모는 단일 관리자 순차 조작 → 실제 동시 충돌 확률은 낮다. 그러나 방어 비용이 거의 0이고, 깨졌을 때 제품 가치가 붕괴하므로 **[지금 필요]**로 강제한다.

**권고 [지금 필요]:**
1. **`UNIQUE(case_id, seq)` 제약을 계약·구현에 명시** — 이것만으로 "조용한 체인 분기"가 "시끄러운 IntegrityError"로 바뀐다(공짜 방어). Evidence 테이블 필수.
2. **채번 직렬화** — 봉인 트랜잭션 시작 시 해당 Case 행을 `SELECT ... FOR UPDATE`로 잠그고 그 안에서 max(seq)+1 채번. Case당 잠금이라 케이스 간 병렬성 손해 없음. (대안: `pg_advisory_xact_lock(case_id)`.) 코드 몇 줄.
3. 자동 봉인은 계약(§3-4-1)대로 **해당 mutation과 동일 트랜잭션**에서 append — `get_db`의 단일 트랜잭션 경계가 이를 자연스럽게 보장하므로 인프라는 이미 적합하다. 이 원자성을 깨는 별도 커밋을 봉인에 넣지 말 것.

> `ponytail:` FOR UPDATE 행잠금이 가장 단순한 정답. 시퀀스 객체·큐·분산락 불필요(YAGNI). 처리량이 문제되면 그때 재검토.

### 🔴-2. `AnomalyExportLog.size_bytes` INT 오버플로우 — 데모값 4.2GB가 int32 초과
**대상:** 보안 §3 `AnomalyExportLog.size_bytes: int`

**근거:** 데모 고정값 4.2GB ≈ **4,509,715,660 bytes**. PostgreSQL `integer`(4바이트) 최댓값 = **2,147,483,647**. 계약 타입 `int`가 PG `integer`로 매핑되면 **삽입 즉시 실패하거나 오버플로우**한다. 즉 데모 시드가 바로 깨진다.

**권고 [지금 필요]:** `size_bytes`는 **BIGINT**(SQLAlchemy `BigInteger`). 바이트 용량은 항상 BIGINT가 정답.
- 파급 점검: `Approval.attachments[].size`(§3-3)도 파일 바이트면 동일 리스크. PDF 수 MB라 int32 내지만, 일관성·안전 위해 BIGINT 권장.

### 🔴-3. FK 컬럼 인덱스 누락 시 전면 풀스캔 — PostgreSQL은 FK 인덱스를 자동 생성하지 않는다
**대상:** 전 계약의 모든 FK — `Item.case_id`·`Approval.item_id`·`Evidence.case_id`·`TradeSecretAsset.case_id`·`AnomalyExportLog.case_id`·`Case.profile_id`·`Case.created_by`·`TemplateItem.rail_template_id`·`Approval.submitter_id`·`Approval.reviewer_id`

**근거:** MySQL과 달리 **PostgreSQL은 FK를 만들어도 참조 컬럼에 인덱스를 자동 생성하지 않는다.** 반면 이 스키마의 지배적 접근 패턴은 전부 FK 기반이다:
- Case 상세: `SELECT * FROM item WHERE case_id=?` (게이트 파생·상태머신)
- 상신 이력: `SELECT * FROM approval WHERE item_id=?`
- 증적 조회: `SELECT * FROM evidence WHERE case_id=? ORDER BY seq`
- 자산/이상반출: `WHERE case_id=?`

인덱스 없으면 전부 **Seq Scan**. 데모 규모(수십 행)에선 체감 안 되지만, 이건 "지금 규모의 성능 함정"이 아니라 **표준 스키마 위생**이며 마이그레이션 한 줄이면 끝난다.

**권고 [지금 필요]:** 모든 FK에 인덱스. 특히 복합으로:
- **`Evidence(case_id, seq)`** — 체인 순서 조회 + 🔴-1 UNIQUE를 겸함(이 UNIQUE 인덱스가 조회 인덱스도 커버).
- **`Item(case_id, status)`** — 게이트 집계(§5)·상태별 count(§3-1-1 `submitted==0`)를 커버.
- 나머지 단일 FK 인덱스는 표준.

## 🟡 성능·타입·제약 — [지금 필요 / Post-MVP 혼합]

### 🟡-1. Gate 파생 집계의 목록 화면 N+1 — [지금 필요]
**대상:** 코어 §5 Gate(비저장·매 조회 계산) + §3-1-1 `review_waiting` 파생 + CM-03 케이스 목록 필터

**근거:** Gate는 저장 엔티티가 아니라 Item 전량 집계다(정당한 설계). Case **상세** 1건 조회는 items 1쿼리로 커버되니 문제없다. 위험은 **케이스 목록(CM-03)에서 케이스마다 완료율/리스크 뱃지/review_waiting을 표시**할 때다. 목록 N건 × (items 로드 + 집계) = **N+1 쿼리**. ORM lazy-load(`case.items`)면 자동으로 이 함정에 빠진다.

**권고 [지금 필요]:**
- 목록 응답에 게이트 파생이 필요하면 **케이스별 반복 금지**. `Item`을 `GROUP BY case_id, status` 한 방 집계 쿼리로 뽑거나, 상세에서만 `selectinload(Case.items)`로 1+1 쿼리. `Item(case_id, status)` 인덱스(🔴-3)가 이 집계를 커버한다.
- 카디널리티: case당 items 십수 개 → 인덱스 집계는 매우 저렴. 데모 규모에서 **머티리얼라이즈드뷰·캐시 테이블 불필요(YAGNI)**.

### 🟡-2. enum을 native PG ENUM으로 만들면 진화가 아프다 — [지금 필요, 구현 결정]
**대상:** 코어 §2 전체 enum(`Rail`·`ExitReason`·`ItemStatus`·`ItemKind`·`StandardTier`·`Role`·`EvidenceEventType` 등) + 레일 로컬 enum(`LaborCaseType` 18종 등)

**근거:** 계약은 확장 예약이 많다 — `ItemKind.external` 예약(§2-4), 레일 추가, `LaborRequiredElement` MVP 후 확장. native PG `ENUM`은 값 추가가 `ALTER TYPE ... ADD VALUE`(구버전 PG는 트랜잭션 내 제약)이고, 값 **삭제·순서변경은 사실상 불가**(타입 재생성). Alembic autogenerate도 native enum 변경을 잘 못 잡는다.

**권고 [지금 필요, 론 재량]:** SQLAlchemy `Enum(..., native_enum=False)` → **VARCHAR + CHECK 제약**으로 매핑. 무결성은 CHECK IN(...)으로 DB가 강제(앱 코드 아님, CLAUDE.md §9 원칙 부합)하면서, 값 추가는 CHECK 재생성만으로 끝나 마이그레이션이 가볍다. MVP 규모에 최적. (성능 차 무의미.)

### 🟡-3. `list[FK]`·`rail_map` json 배열 = FK 무결성 DB 강제 포기 — [Post-MVP, 단 명시 필요]
**대상:** `Item.standard_ids`·`TemplateItem.standard_ids`(list[FK→Standard]) / `Profile.rail_map`(json{rail→rail_template_id}) / TS `Item.detail.related_asset_ids`

**근거:** FK를 json 배열/객체에 담으면 **참조 무결성을 DB가 강제하지 못한다.** `Standard`/`RailTemplate`이 삭제·교체돼도 배열 안 id는 dangling으로 남고, "이 Standard를 참조하는 Item" 역조회도 조인으로 못 한다. 정규형은 조인 테이블(`item_standard` M:N, `profile_rail_template`).

**MVP 판단:** `Standard`·`RailTemplate`은 **읽기 전용 시드**(삭제 안 함)이고 역조회 수요가 없다. 따라서 json 배열은 **실용적으로 허용 가능한 비정규화**다. 단 "FK 무결성을 DB가 아니라 시드 관리로 보증한다"는 사실을 **명시적으로 인정하고 넘어갈 것**(모르고 지나가면 나중에 dangling 버그).

**권고:** [Post-MVP] 회사 커스텀 템플릿·Standard 편집이 생기면 조인 테이블로 정규화. MVP는 json 배열 유지하되 시드 무결성 책임을 문서화.

### 🟡-4. `Case.status` 이중 진실(저장 vs 계산 "택1") — desync 위험 — [지금 필요, 구현 결정]
**대상:** 코어 §3-1-1 "review_waiting … 저장하거나 계산 중 택1, 진실은 항목 상태"

**근거:** 계약이 진실 소재를 열어뒀다. **저장**을 택하면 항목 전이 시 case.status를 함께 갱신하지 않는 순간 desync(항목은 submitted인데 목록엔 in_progress). **계산**을 택하면 목록 필터마다 항목 집계.

**권고 [지금 필요, 론 재량]:** 하나로 고정하라.
- 목록 필터(CM-03) 성능을 위해 **저장 + 항목 전이와 동일 트랜잭션에서 재계산·갱신**(`Case.recompute_status()`를 Approval confirm/reject 경로에서 항상 호출) 권장. §3-1-1이 이미 이 트리거를 정의하고 있으니 그 경로를 단일 진입점으로.
- desync 안전망: 목록 조회 시 status 필터는 저장값 사용하되, 상세에서는 항목 기반 재계산으로 교차검증(개발 중 assert).

### 🟡-5. FK `ON DELETE` 정책이 계약 전무 — 증적 CASCADE 삭제는 재앙 — [지금 필요/결정필요]
**대상:** 전 계약 FK. 특히 `Evidence.case_id`·`TradeSecretAsset.case_id`·`AnomalyExportLog.case_id`

**근거:** 계약 어디에도 삭제 정책이 없다. `Evidence`는 append-only 봉인 증거(§3-4 "변경불가")인데, `Case`에 `ON DELETE CASCADE`가 걸리면 **케이스 삭제 시 법적 증거가 물리 소실**된다 — 봉인의 의미가 사라진다. 반대로 아무 정책이 없으면 기본 `NO ACTION`이라 Case 삭제가 FK 위반으로 막힌다(실은 이게 증거 보호엔 안전).

**권고:**
- **[지금 필요]** `Evidence`(및 봉인 대상 로그)는 **절대 CASCADE 금지.** Case는 물리 삭제 대신 **soft-delete**(상태 플래그) 또는 삭제 자체를 막는 게 제품 성격에 맞다.
- `Item`·`Approval`은 Case에 `ON DELETE CASCADE` 후보(케이스 없으면 무의미). 단 Case를 soft-delete로 가면 CASCADE 자체가 불필요.
- **[결정필요]** "Case 삭제를 허용하는가, 한다면 증적은 어떻게 보존하는가"는 제품 정책 → 헤르미온느·도비 판단 대상(§아래).

### 🟡-6. json이 아니라 jsonb를 써라 — [지금 필요, 구현]
**대상:** `Evidence.payload`·`Item.detail`·`Profile.rail_map`·`Approval.attachments` 등 모든 json 컬럼

**근거:** PostgreSQL에서 `json`은 텍스트 원문 보존(파싱 없음), `jsonb`는 이진 저장으로 연산·인덱싱·중복키 정규화가 우월하다. 단 canonical 직렬화 순서 보존이 필요한 곳은 주의:
- **`Evidence.payload`는 주의** — 🔴-1의 `integrity_hash = SHA-256(canonical_json(payload)+prev_hash)`는 **애플리케이션이 canonical_json 문자열을 만들어 해시**하는 것이 계약이다. 해시 입력 문자열은 앱이 통제하므로, 저장 컬럼이 json이든 jsonb든 **해시 재현성과 무관**하다(저장값에서 재직렬화해 해시하지 말 것 — 봉인 시점 canonical 문자열 자체를 해시 입력으로 확정). 저장은 조회 편의상 jsonb 권장.
- 나머지 `detail`·`rail_map`은 jsonb 권장(향후 `->>'key'` 필터·GIN 여지).

## 🔵 개선·확장성 — [Post-MVP]

### 🔵-1. PK 타입 int vs uuid 미정 — MVP는 BIGINT로 충분 — [Post-MVP]
계약 `id: int/uuid` 미결. MVP는 단일 PostgreSQL·데모 시드 → **BIGINT identity PK**가 가장 단순하고 증적 `seq`와도 자연 정합(YAGNI: uuid 불필요). 단 `GET /cases/{id}`처럼 순차 id가 URL 노출돼 추측 가능성이 우려되면 `Case`만 uuid 고려. 구현 재량.

### 🔵-2. Evidence의 `updated_at`은 의미 없음(append-only) — [Post-MVP/개선]
CLAUDE.md §9 `TimestampMixin`(created/updated)을 Evidence에 그대로 붙이면 불변 레코드에 `updated_at`이 무의미. `sealed_at`이 진짜 시각. Evidence는 Mixin에서 updated_at 제외하거나, 물리적 불변 보장(Post-MVP: DB 트리거로 UPDATE/DELETE 차단 — MVP엔 앱 레벨 append-only + 🔴-1 UNIQUE로 충분, 트리거는 과함).

### 🔵-3. `LaborPrecedent` 코퍼스 인덱스 — [Post-MVP]
조회 패턴 `WHERE 'written_notice' ∈ matched_elements AND category ∈ 해고계열`. **399행**이라 Seq Scan도 즉시 끝난다 → 지금 인덱스 불필요. Post-MVP로 데이터가 커지면 `category` B-tree + `matched_elements`(PG array라면) GIN. `seq`(CSV 순번)는 참조 조회키라 UNIQUE 인덱스 정도만 저렴하게.

### 🔵-4. `matched_elements`·`granted_scopes` 등 배열 컬럼 — PG array vs json — [Post-MVP]
`list[enum]`/`list[str]` 저장은 PG **native array + GIN**이 컨테인먼트 쿼리(`@>`)에 유리하나, MVP 규모에선 jsonb·풀스캔으로 충분. 구현 편의대로.

## 인프라(B) 개별 코멘트

- **`db.py` — 깨끗하다.** async 엔진(`pool_pre_ping`·`pool_recycle=300`), `expire_on_commit=False`, 요청=트랜잭션 경계(성공 commit/예외 rollback)가 CLAUDE.md §5와 정확히 일치. 🔴-1의 "봉인=같은 트랜잭션" 원자성을 이 경계가 **공짜로 보장**한다. 좋다.
- **`NAMING_CONVENTION` — 표준 권장형.** 다만 `uq`/`fk`가 `column_0_name`이라 **복합 제약에서 이름 충돌 여지**가 있다. 예: `Evidence`에 `UNIQUE(case_id, seq)`를 걸면 이름이 `uq_evidence_case_id`가 되어 case_id 관련 다른 uq와 충돌 가능. 실무 위험 낮음 — [Post-MVP] 충돌 나는 제약만 `name=`으로 명시 지정. `ix`의 `column_0_label`은 OK.
- **`config.py` — 두 가지 짚음:**
  - **[Post-MVP]** `database_url` 기본값에 크리덴셜(`exitguard:exitguard`) 하드코딩. 로컬 데모용이라 실위험 낮으나, 운영 URL 미설정 시 **조용히 로컬 데모 DB로 붙는** CLAUDE.md §8("위험 설정 기동 validator 차단") 위반 소지. 데모 단계는 허용.
  - **[지금 필요]** CLAUDE.md §16이 요구하는 **테스트 DB 분리 가드(`DB_NAME.endswith("_test")`)가 config에 없다.** 통합 테스트(repository+실 테스트DB)를 짜기 전에 이 가드를 넣어야 **테스트가 개발 DB를 밀어버리는 사고**를 막는다. 다음 라운드(models/repository) 전에 선반영 권고.
- **`alembic/env.py` — 깨끗하다.** async 배선·단일 원천 URL·`target_metadata=Base.metadata` 정상. `Base.metadata`에 도메인 모델이 import되어 등록되는지만 다음 라운드에서 확인(모델 파일이 `env.py`에서 import 경로로 로드돼야 autogenerate가 잡는다).

## 의도적 비정규화 인정 (정당 — 유지)

1. **`Evidence.payload` 스냅샷 봉인(§3-4-1)** — 처리 시점 사실을 중복 저장. **재현성·위변조 탐지** 목적. 정규화보다 우선. **정당·인정.**
2. **compare 결과의 증적 편입(`compare_recorded`) → `DefenseReport.compare_findings`(§10)** — 봉인된 CompareResult를 재계산 없이 인용. `Standard.version`이 이후 올라가도 봉인 시점 원문이 보존돼야 하는 **법적 증거 재현성** 요구. 실시간 조인 대신 스냅샷이 **정답**. **정당·인정.**
3. **Gate 비저장 파생(§5)** — 저장 중복 대신 결정론적 계산을 진실로 둠. 정규화 관점에서 오히려 모범(파생값 저장 이상 회피). 성능은 🟡-1 인덱스로 커버. **정당·인정.**
4. **`Item.detail`(jsonb) 레일 확장 슬롯(§7)** — EAV 안티패턴 경계는 있으나, 조회 필터 대상이 아니라 레일별 표시·파생용 예약 슬롯이라 **스키마 유연성 목적의 허용 가능한 비정규화.** jsonb 권장(🟡-6).

## [결정필요] — 도비 → 헤르미온느/론 조율 대상

1. **Case 삭제 정책 & 증적 보존(🟡-5).** "Case 물리 삭제를 허용하는가? 허용 시 봉인 Evidence는 어떻게 보존하는가?" 계약에 삭제 정책이 전무하다. 증적 CASCADE는 제품 가치와 충돌 → **soft-delete 또는 삭제 금지**를 계약에 명문화할지 헤르미온느 판단 필요. (스네이프 권고: 증적은 물리 삭제 불가로 고정.)
2. **`UNIQUE(case_id, seq)` + seq 채번 직렬화(🔴-1)를 계약 §3-4-1에 명시할지.** "seq 단조증가·누락 없는 append"는 이미 계약이나, **동시성 보장 수단(UNIQUE 제약·행잠금)** 이 계약에 없다. 무결성 핵심이라 계약 레벨에 한 줄 추가 권고 → 헤르미온느 확정 후 론 구현.
3. **`size_bytes` 등 바이트 타입을 계약에서 BIGINT로 못박을지(🔴-2).** 계약 표기 `int`가 구현자에게 int32 오해를 줄 수 있으니, 코어 §0 범례에 "바이트/대용량 정수는 64-bit"를 한 줄 추가 권고.

> 그 외(enum 매핑·jsonb·PK 타입·인덱스)는 **구현(론) 재량 권고**라 계약 변경 없이 다음 라운드 models.py 감수에서 확인한다.
