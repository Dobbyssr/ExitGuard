# Phase 1 — 각 대표 작업 가이드 (레일별 계약 살 붙이기)

> **전제**: Phase 0 공통 뼈대 3장이 freeze 됨 — `유비쿼터스.md` · `data-model.md` · `api-spec.md`.
> 이 3장은 **읽기만** 한다. 여기 든 `Case`·`Item(검사항목)`·`Gate`·`compare shape`·enum·상태머신은 **절대 다시 정의하지 않는다**(import).
> 범위 = **CP 23개 우선**. 나머지 Post-MVP는 다음 라운드.

---

## 0. 셋 다 지키는 황금 규칙 (병렬이 안 부딪히는 이유)

1. **뼈대 재정의 금지.** 공유 엔티티·enum·상태·compare shape는 뼈대에서 가져다 참조만.
2. **레일 고유는 딱 3곳에만 붙인다.** ① `Item.detail`(레일 필드) ② `compare.py` 레일 *규칙*(shape는 고정) ③ `Standard` 레일 근거 시드.
3. **데모(`pitch/데모/ExitGuard-나모업로드-단일본.html`)가 1차 진실.** 항목코드·자산 13건·이상반출 4.2GB 등 수치는 데모에서 그대로.
4. **직역법 §4.** compare `risk` 행은 "…로 판정된 **사례가 있습니다**" 프레이밍, `boundary_notice`는 고정 문구. "진단·위법·판단" 금지.
5. **폴더 = owner = git 충돌 없음.** 각 대표는 **자기 레일 폴더만** 생성/수정. 뼈대 3장(루트)은 안 건드림(수정 필요 시 도비에 요청).

**디렉토리 구조** (레일=폴더 하나=대표 한 명):
```
docs/spec/
  유비쿼터스.md · data-model.md · api-spec.md   ← 뼈대(공유·읽기전용, 루트)
  기능리스트.md · PHASE1-가이드.md
  공통/     기능명세서.md                         ← 대표1
  노무/     기능명세서.md · data-model.md · api-spec.md   ← 대표2
  영업비밀/ 기능명세서.md · data-model.md · api-spec.md   ← 대표3
  보안/     기능명세서.md · data-model.md          ← 대표3
```
(빈 폴더 미리 안 만들어도 됨 — 헤르미온느가 파일 생성 시 자동 생성)

각 대표가 만드는 산출 = **① 기능명세서(동작) ② data-model 레일 보강 ③ api-spec 레일 엔드포인트** (대표1은 ① 위주 — 코어 계약은 Phase 0에서 끝).

## 대표2 — 노무 LB

**담당 CP (5)**: LB-01·02·03·04·08
**산출**: `docs/spec/노무/` 폴더 — `기능명세서.md` + `data-model.md`(레일 보강) + `api-spec.md`(레일 엔드포인트)

**작업**:
1. **기능명세서**: LB-01 법정기한 타임라인 / LB-02 D-day 관제·알림 / LB-03 6항목 상태 대조 / LB-04 판정례 대조 / LB-08 근거 카드
2. **data-model 노무 보강**: `Item.detail`(rail=labor) 스키마 = 기한요소·정산 필드. 항목코드 **L-01~L-09** 정의(사직합의서·연차수당·금품청산14일·이직확인서·4대보험·해고예고30일 등).
3. **법정기한 계산식(결정론적)**: `exit_date` 기준 각 기한 D-day 산출 규칙 명세(LB-01/02).
4. **compare 규칙(LB-04)**: 노무 신호(예: "구두 해고→서면통지 누락") → L2 판정례 요소 매핑. shape는 §6 고정, *규칙만* 채움.
5. **근거 시드**: `Standard` 노무 L1(근기법 §36·§27·§26)·L2(중노위·대법원)·L3(고용부).
6. **api-spec**: `GET /cases/{id}/rails/labor` 내부 응답(타임라인·D-day) 채움.

**착수(자기 세션에서 도비에게)**:
> `Phase 1 노무 LB 착수. docs/spec/PHASE1-가이드.md 대표2 섹션대로 헤르미온느에 위임해줘.`

## 대표3 — 영업비밀 TS + 보안 SEC

**담당 CP (5)**: TS-01·02·05 · SEC-01·02
**산출**: `docs/spec/영업비밀/`(기능명세서·data-model·api-spec) + `docs/spec/보안/`(기능명세서·data-model)

**작업 — 영업비밀 TS**:
1. **기능명세서**: TS-01 접근 자산 자동 특정 / TS-02 보호요건 3요소 대조 / TS-05 비밀관리성 대조
2. **data-model TS 보강**: 자산 인벤토리(**13건**). `Item.detail`로 부족하면 **별도 레일 테이블 신설 허용**(단 코어 FK·상태·증적 규약 준수). 보호요건 3요소(비밀표시·재서약·접근제한).
3. **compare 규칙(TS-05)**: 비밀관리성 패턴("비밀표시 없음+전원 접근") → L2 판례 요소 매핑.
4. **근거 시드**: `Standard` TS L2(판례)·L3.
5. **api-spec**: `GET /cases/{id}/rails/trade_secret` 내부.

**작업 — 보안 SEC** (대부분 **시뮬** → 가볍게):
1. **기능명세서**: SEC-01 계정·SaaS·기기 회수 체크리스트(상신→확인) / SEC-02 이상반출 이력(**4.2GB** 데모값)
2. **data-model SEC 보강**: 회수항목 `Item.detail`(시뮬) + 이상반출 로그(시뮬).
3. **api-spec**: `GET /cases/{id}/rails/security` 내부. **경계**: "확인 기록이지 회수 실행 아님"(SEC-04) 명시.

**착수(자기 세션에서 도비에게)**:
> `Phase 1 영업비밀+보안 착수. docs/spec/PHASE1-가이드.md 대표3 섹션대로 헤르미온느에 위임해줘.`
