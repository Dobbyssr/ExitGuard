## 공통 전제 (LB 5기능 공통)

- **레일**: `labor`(LB · 항목코드 `L-`).
- **액터**: 인사총무(오케스트레이터/허브 = `admin`, 데모 "한지수") · 담당자(노무 처리 = `user`, 대리 상신). 대상자(퇴사자)는 데이터(Case).
- **구현등급**: LB-01 `[진짜구현]`(결정론적 기한 계산) · LB-02 `[진짜구현]`(D-day 파생) · LB-03 `[진짜구현]`(상태 대조 파생) · LB-04 `[진짜구현]`(AI 판정례 대조 = **해자**) · LB-08 `[진짜구현]`(근거 배지). **모두 진짜구현** — 시뮬 입력 없음(노무는 회사 입력·공개기준 대조가 전부).

## LB-01 ★CP — 유형별 법정 기한 타임라인

**목적**: 퇴사 유형에 맞춰 금품청산 14일(§36)·해고예고 30일(§26)·서면통지(§27)·이직확인서·4대보험 상실·연차 정산 기한을 **결정론적으로 자동 배치**한다.

**액터**: 인사총무 · (시스템)

**사전조건**: 케이스 접수 완료(CM-04) · 프로파일 적용으로 노무 `Item` 6개 생성(코어 `Case.apply_profile()`).

**트리거**: 레일상세(CM-07)에서 노무 탭 진입 / `GET /cases/{id}/rails/labor`.

**기본 플로우**:
1. 케이스의 노무 `Item` 6개(L-01·02·04·06·08·09)를 조회한다.
2. 각 항목의 `detail.deadline_rule`(data-model §2-2)로 `Item.deadline`을 **결정론적 계산**한다:
   - `settlement_14d`(L-02·L-04) → `deadline = exit_date + 14d` (§36 실추출).
   - `dismissal_notice_30d`(L-09) → 권고사직이므로 **`not_applicable`**(해고 아님, 코어 §4).
   - `written_notice`(서면통지) → 기한(일수) 아닌 **이벤트 요건** → LB-04 대조 요소로 넘김.
   - `separation_cert`(L-06)·`insurance_loss`(L-08) → `[시드확인필요]`(조문 기한 미실측) → "발급/신고 필요"로만 배치, D-day 공란.
3. `timeline_group`(settlement/notice/filing)으로 묶어 타임라인 렌더.

**입력**: `Case.exit_date`·`exit_reason` + 노무 `Item[]`. **출력**: 기한 배치된 `items[]`(각 `deadline`·`detail`) — `GET /cases/{id}/rails/labor` 응답.

**데이터**: 코어 `Item` + `Item.detail`(data-model §2-2). `LaborDeadlineRule` enum.

**상태전이**: 없음(기한 파생·배치). 항목 처리는 코어 상신-검토 상태머신.

**수용 기준(AC)**:
- AC-1: `settlement_14d` 항목의 `deadline == exit_date + 14일`(§36 근거 표기).
- AC-2: `exit_reason=recommended_resignation`일 때 L-09(해고예고)가 `not_applicable`.
- AC-3: `separation_cert`·`insurance_loss`는 임의 일수 없이 `[시드확인필요]`(D-day null) — **창작 금지 준수**.
- AC-4: 각 기한 항목에 근거 조문(`deadline_basis`, 예: "근로기준법 제36조") 표기.

**문구경계**: "법정 기한 타임라인"·"자동 배치" = 공개 조문의 결정론적 계산. 어느 것이 "위반"인지 **판정하지 않음**(그건 D-day 관제·사례 대조·전문가 몫).

## LB-02 ★CP — D-day 기한 관제·알림

**목적**: 각 법정 기한의 잔여일(D-day)을 계산해 임박·경과를 관제·알림한다(예: 금품청산 지급 예정일 D-3 근접).

**액터**: 인사총무

**사전조건**: LB-01 기한 배치 완료(`Item.deadline` 존재).

**트리거**: 노무 탭 렌더 / 대시보드 KPI / 알림(CM-28 연계, MVP는 표시).

**기본 플로우**:
1. 기한 있는 각 항목의 `dday = deadline − reference_date`를 계산(data-model §2-2, 파생·저장 아님).
2. 상태 구간 표시: `D-n`(임박·잔여) / `D-day`(당일) / `D+n`(경과). 임박 임계는 표시용(예: D-3 이하 강조).
3. 금품청산(L-04) 등 blocking 항목의 D-day를 게이트/대조 `status` 행에 반영.

**입력**: `Item.deadline` + reference_date(today). **출력**: 항목별 `detail.dday` + 구간 라벨.

**데이터**: 파생값(`dday`). 저장 아님.

**상태전이**: 없음.

**수용 기준(AC)**:
- AC-1: `dday`가 `deadline − reference_date`로 결정론적 계산(음수=경과).
- AC-2: 금품청산(§36) 지급기한 D-day가 LB-04 compare `status` 행 수치와 일치.
- AC-3: `[시드확인필요]` 기한(이직확인서·4대보험) 항목은 D-day를 산출하지 않음(null).

**문구경계**: "기한 관제 / D-3 / 임박 / 경과" = 시간 대조. "지급하지 않으면 위법" 등 단정·개선방안 어조 금지(사실=기한·잔여일만).

## LB-03 ★CP — 노무 검사 항목 상태 대조

**목적**: 6개 노무 항목을 회사 입력 상태와 공개 기준으로 대조해 **충족/미충족/해당없음**으로 표시하고, 명시적 count로 요약한다.

**액터**: 인사총무

**사전조건**: 노무 `Item` 6개 존재(상태 상신-검토로 갱신).

**트리거**: 노무 탭 렌더 / 상태 변경(상신·검토) 후 재파생.

**기본 플로우**:
1. 6개 항목의 코어 `Item.status`를 3-상태로 매핑(data-model §3-1): `approved`→충족 / `not_applicable`→해당없음 / `{pending,submitted,rejected}`→미충족(하위상태 표기).
2. **명시적 count 파생**(data-model §3-2): `met_count`·`na_count`·`unmet_count`(불변식 `합 == total`).
3. 항목 상태 배지 테이블 렌더 + 요약("충족 N · 미충족 N · 해당없음 N").
4. 미충족 항목은 코어 상신-검토(CM-10)로 보완 진입.

**입력**: 노무 `Item[].status`. **출력**: 3-상태 배지 + 요약 count(`GET /cases/{id}/rails/labor` 응답 내).

**데이터**: 코어 `Item.status`. 파생식 data-model §3.

**상태전이(항목 갱신)**: 코어 상태머신(§4) 그대로 — 상신(pending→submitted)·확인(→approved)·반려(→rejected). LB는 그 결과를 3-상태로 뷰잉만.

**수용 기준(AC)**:
- AC-1: 데모 6항목 상태 조합이 §3-3 캘리브레이션 표대로 `met=2·na=1·unmet=3`을 산출(불변식 `2+1+3=6` 통과).
- AC-2: 레일 완료율이 코어 §5식으로 `round(100*2/5)=40%`.
- AC-3: `submitted` 항목은 "미충족(상신검토필요)" 하위상태로 구분 표기(단순 "미충족" 단정 회피).
- AC-4: 상신→확인 시 해당 항목이 충족으로 재파생되고 count·완료율이 갱신됨.

**문구경계**: "충족/미충족/해당없음" = **상태 대조 결과**. "미충족 = 위법" 단정 금지. **보완항목은 기존 템플릿 항목 연결이지 생성형 개선방안 제시가 아님**(노무관리진단 정의 회피 — PRODUCT §4).

## LB-04 ★CP — 공개 판정례 대조 알림 (= 기술 해자)

**목적**: 케이스 사실(회사사유 텍스트)에서 절차 리스크 신호를 추출해 중노위 공개 판정례의 요구 요소와 대조하고, "누락 시 …로 판정된 사례가 있습니다"를 사례 프레이밍으로 알린다.

**액터**: 인사총무

**사전조건**: 케이스 접수(reason_text 존재) · `Standard` L1/L2 시드 · `LaborPrecedent` 코퍼스 적재.

**트리거**: 노무 탭 진입 시 자동 대조 / 항목 드로어(CM-09) 근거 조회 / 인테이크(CM-05)와 공용 엔진.

**기본 플로우**:
1. `case_facts.reason_text`에서 **비서면 통보 신호**(구두·문자·전화 등) 스캔(data-model §5-2).
2. 신호 → `written_notice`(서면통지 §27) 요구 요소 매핑 → 해당 요소를 커버하는 `LaborPrecedent`(해고계열) 필터.
3. 코어 `POST /compare`(rail=labor) 호출 → `CompareResult` 5행 생성(data-model §5-3).
4. 5행 표시: 절차(신호 요약) · 기준(§27 인용) · 위험(판정례 사례 프레이밍, 순번 51·388) · 상태(미충족 3건·금품청산 D-day) · 출처(링크).
5. 근거 배지 L1(§27)·L2(중노위 판정례)·L3(고용부) + `boundary_notice` 표시. 판단 필요 시 [전문가 연결].

**입력**: `CompareInput`(rail=labor). **출력**: `CompareResult`(코어 §6-2, LB 규칙 data-model §5).

**데이터**: `Standard` L1/L2/L3 + `LaborPrecedent`(순번 51·388) + compare 규칙 §5.

**상태전이**: 없음(조회·대조). 대조 결과는 증적 봉인(CM-12) 대상.

**수용 기준(AC)**:
- AC-1: `rows`가 정확히 5행, kind 순서 `procedure→standard→risk→status→source` 고정.
- AC-2: `risk` 행이 "구두·약식 통보…**부당해고로 판정된 사례가 있습니다**(중앙노동위 판정례: 문자 통보→서면통지 위반)" 사례 프레이밍(**단정 금지**).
- AC-3: `boundary_notice`가 코어 §6-3 고정 문구와 **글자 그대로** 일치.
- AC-4: `badges`에 L1·L2·L3 3층 전부 포함(원천 링크).
- AC-5: `status` 행 "미충족 3건"이 LB-03 `unmet_count`와, 금품청산 D-day가 LB-02와 일치.
- AC-6: **§36(금품청산) 판정례 risk 행을 만들지 않는다**(코퍼스 0건 — data-model §4-2). §36은 status D-day로만.
- AC-7: 인용 판정례에 정식 사건번호가 없으면 사건번호를 **표기하지 않는다**(순번 51·388 = case_no null, 창작 금지).

**문구경계**:
- ✅ 허용: "…부당해고로 판정된 **사례가 있습니다**" · "구두·문자 통보 정황 신호" · "미충족 3건" · "서면통지 요구 요소".
- ❌ 금지: "부당해고입니다"(단정) · "위법입니다" · "패소합니다" · "진단" · "법률 자문" · "실질적 해고다"(판단).
- LLM은 공개 기준 **인용·링크 위주**. 생성형 법률 판단·개선방안 제시 배제. 권고사직↔해고 성격 판정은 **하지 않고** "다퉈질 경우"의 대조·사례만.

## LB-08 ★CP — 기준 대조 근거 카드

**목적**: 노무 레일의 모든 대조·항목에 L1/L2/L3 근거 배지를 붙이고 "위법 여부 판정 안 함" 고지를 명시한다("GPT 래퍼 아님" 증명).

**액터**: 인사총무 · 전원

**사전조건**: `Standard`(rail=labor) L1/L2/L3 시드(data-model §6).

**트리거**: 노무 탭·항목 드로어·compare 결과·리포트 어디서든 근거 표기 지점.

**기본 플로우**:
1. 대조/항목/리포트 응답의 `badges`(코어 §1-6 규약)에 L1(근기법 §26·§27·§36)·L2(중노위 판정례)·L3(고용부 가이드) 배지 렌더.
2. 각 배지: `{tier, title, source_url, version}`. tier로 색/라벨 렌더.
3. **고정 고지** 노출: "본 서비스는 위법 여부를 판정하지 않으며, 공개된 법령·판례 기준과 회사 입력 상태의 대조 결과·기한만 제공합니다."(= boundary_notice 계열)
4. 판단 분기에 [전문가 연결] 버튼.

**입력**: `Standard[]`(rail=labor). **출력**: `badges[]` + 고지 문구(모든 노무 대조 응답 공통).

**데이터**: 코어 `Standard`(data-model §6). `GET /standards?rail=labor`.

**상태전이**: 없음.

**수용 기준(AC)**:
- AC-1: 노무 대조·항목 응답에 L1·L2·L3 배지가 누락 없이 포함(대조결과 배지 필수 — CM-15).
- AC-2: L1 배지가 근기법 §26·§27·§36 중 해당 조문을 원문 링크(law.go.kr)로 표기.
- AC-3: "위법 여부 판정 안 함" 고지가 노무 레일 화면에 상시 노출.
- AC-4: L3 구체 문서 미확정 시 `[시드확인필요]`로 두되 L1·L2만으로 카드 성립.

**문구경계**: 근거 카드는 **출처 표기·경계 고지** 전용. 카드 자체가 "판단"을 서술하지 않는다. "진단 결과" 류 라벨 금지.

## (얇은 참조) LB-05 · LB-06 — 코어 워크플로우로 흡수
