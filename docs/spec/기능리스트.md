# ExitGuard 기능 리스트 (데모 기준 재작성)

> **문서 성격** — ExitGuard **실제 데모 앱**에 존재하는 화면·기능을 빠짐없이 도출하고, 그 위에 PRODUCT.md 방향(고칠 것/고도화할 것)을 델타로 얹어 분류한다. 리스트 레벨(기능ID·이름·1줄목적·액터·태그)까지만 다룬다. 각 기능의 상세 플로우·상태전이·수용기준은 이 리스트 확정 후 Feature Spec으로 별도 명세한다.
> **근거 SSOT** — **데모(`pitch/데모/ExitGuard-나모업로드-단일본.html`) 기준 (base = 진실)** + `dobby/PRODUCT.md`(§1·§3·§4·§6) 델타. 데모에 없는 사실은 창작하지 않음(PRODUCT 델타는 `[신규]`로 표기).
> **작성**: 헤르미온느(기획·명세) · **작성일**: 2026-07-14 (데모 전량 정독 기준 전면 재작성)

## 0. 범례 (반드시 먼저 읽을 것)

### 기능 ID 접두어 (3레일 + 공통)

| `CM` | 공통(Common) | 셸·내비·대시보드·인테이크·게이트·증적·상신검토·템플릿·기준DB·전문가·설정·UX |

### 태그 1 — 출처·변경 성격 (신규 축, 전 기능 필수)
- `[데모유지]` = 데모에 있고 본선에서 그대로 감
- `[데모수정]` = 데모에 있으나 PRODUCT 방향대로 고침 (무엇을 왜 고치는지 1줄)
- `[고도화]` = 데모에 있으나 본선에서 수준을 올림 (목업 정적 → AI 실동작 / 결정론적 계산)
- `[신규]` = 데모엔 없고 PRODUCT.md에만 있는 것 (왜 필요한지 1줄)

### 태그 2 — 구현 등급 (PRODUCT §6 경계)
- `[진짜구현]` = 본선에서 실제 동작. 대상 = **AI 판례·판정례 대조 엔진, 3층 기준 DB, 결정론적 기한 계산, 게이트 집계, 증적 봉인·Export, 상신-검토 워크플로우** (= 유일한 기술 해자).
- `[시뮬데이터]` = 고정 시뮬레이션 데이터로 대체. 대상 = **DLP/오피스키퍼 로그, 계정·SaaS 회수 실행, 그룹웨어 자동연동, 권한/자산/계정 원천 데이터**. **여기 리소스 최소화.**

### 태그 3 — MVP 포함 / ★CP
- `[MVP]` = 세로절단 시나리오에 필요 / `[Post-MVP]` = 이번 6주 범위 밖.
- **★CP (Critical Path)** = 세로절단 시나리오("핵심 개발자 **김민수** 퇴사 1건 → 3레일 관통 → '방어 가능 상태' 리포트 + 증적 Export")가 처음부터 끝까지 돌기 위한 최소 필수 기능. (§6 별도 정리)

### 근거 배지 (판정 요소가 있는 기능만)
- **L1 법령**(국가법령정보센터) / **L2 판례·판정례**(대법원·중노위) / **L3 정부가이드**(고용부·특허청 매뉴얼·표준서식). PRODUCT §3-2 3층 기준 스택.

### 액터
- **인사총무**(오케스트레이터/허브, 데모의 "한지수 · 인사담당 · 관리자") · **대상자**(퇴사자 본인·레일 담당자 — 데모의 "대상자 상신" 주체) · **관리자**(상신 검토·승인). PRODUCT §3-4 오케스트레이터 모델.
- 인증은 **admin/user 2역할**로 단순화. 데모는 관리자(한지수) 로그인 상태로 시작한다.
- **⚙️ 확장성 주석**: 영업비밀·보안 레일 담당 역할은 데이터모델에서 분리 가능하게 남긴다(대기업 3팀 분리 대응).

### ⚖️ 문구 규율 (이 문서 전체 적용 — PRODUCT §4)
금지어("진단"·"위법입니다"·"부당해고입니다"·"패소합니다"·"노무상담/법률자문"·AI 생성형 법률판단) 미사용. 판정 표현은 **"상태 대조 / 기한 관제 / 리스크 알림 / 기준 대비 미충족 N건 / 공개 판정례에서 이 요소 누락 시 …로 판정된 사례가 있습니다"** 프레이밍으로만 기술. (데모의 "위반 리스크"·"AI 답변" 등 경계 표현은 §8 델타 로그에 교정 기록)

- `CM-24`(인증) — 데모가 관리자 로그인 상태로 시작하므로 pierce-flow 구동에 load-bearing 아님. 데모를 로그인 화면부터 시연하기로 정하면 CP 승격(§7-3).
- `CM-11`(일괄 상신)·`CM-14`(아카이브)·`CM-19`(전문가) — MVP지만 세로절단 단일 케이스 관통에는 CM-10/CM-12/CM-13로 충분.

## 8. 데모 → 본선 델타 로그

### (a) 데모에 있는데 이전 리스트(36개)가 놓쳤던 화면/기능
1. **상신-검토 워크플로우(CM-10)** — 대상자 상신 → 관리자 검토(확인완료/반려·재상신), 전자서명·첨부·기준대조. 데모의 **핵심 상호작용**인데 이전 리스트엔 상태 흐름 자체가 없었음.
2. **일괄 상신(CM-11)** — 대상자별 다중 항목 일괄 상신.
3. **검사 항목 상세 드로어(CM-09)** — 항목별 법령·확인요건·리스크.
4. **직종·직급 템플릿(프로파일) + 레일 템플릿 2계층(CM-20·21·22)** — 이전 리스트의 L1/L2/L3 추상 모델과 실제 데모 구조(레일 템플릿=항목 라이브러리, 직종·직급 프로파일=매핑, 케이스=자동배정)가 다름. 데모 구조로 재정렬.
5. **법률 기준 DB 개요 + 레일 트리(CM-16·17)** — 소스 버전카드·갱신이력·법령 트리·검색·페이지네이션.
6. **오프라인/폐쇄망 기준 DB 업데이트(CM-18)** — .egpkg 서명패키지 검증·적용. (신규)
7. **레일 항목 추가 AI 검토(CM-23)** — 근거문서 첨부 → AI 분류·대조규칙 생성.
8. **조직·직급 체계 화이트라벨(CM-29)** — 용어 커스터마이징·조직도·직책별 템플릿 연결.
9. **설정 6종(CM-25~29)** — 조직정보·자동수집·시스템연동·알림·조직도.
10. **UX 인프라(CM-30~32)** — 커맨드 팔레트 ⌘K·다크모드·토스트.
11. **케이스 접수 3경로 + AI 대조 패널(CM-04·05)** — 이전 리스트는 "대화형 입력"으로 단순화했으나 데모는 폼+AI 대조.
12. **증적 봉인 뷰 SHA-256 무결성·처리이력 표(CM-12/M5)** — 봉인의 구체 형태.

### (b) 데모수정·고도화 항목과 이유
- **CM-04 [데모수정]**: 이전 "대화형(챗) 입력" → 데모의 **폼+AI 대조 패널**로 교정. 대화형은 데모 근거 없음(창작 금지). → §7-1 결정필요.
- **CM-24 [데모수정]**: 데모는 인증 화면 없이 관리자(한지수) 로그인 상태로 시작 → 인증은 MVP지만 데모 연출상 생략 가능.
- **[고도화] 공통**: 데모의 정적 %·목업(게이트 집계·기한 계산·판례 대조·증적 봉인·PDF)을 본선에서 **실제 동작**으로. AI 판례·판정례 대조(LB-04·TS-05)·3층 기준 DB(CM-16)가 해자의 핵심 = 반드시 진짜구현.

### (c) 데모엔 있으나 MVP에서 뺀 것 (Post-MVP)
- CM-17 법령 트리, CM-18 폐쇄망 업데이트, CM-20·21·23 템플릿 편집/AI검토, CM-25~31 설정·UX 대부분, LB-07, TS-06, SEC-05·06. → 세로절단 관통에 불필요. 데모 화면은 유지하되 데이터는 고정.

### (d) 직역법 교정 건 (데모 문구 → 허용 프레이밍)
- 데모 전체 **"진단" 0건** — 통과. ✅
- **"위반 리스크"**(검사 항목 상세 드로어 헤딩) → **"미충족 리스크 / 리스크 알림"** 으로 교정 권고. ("위반" 단정 어감 회피)
- **"AI 답변은 참고용입니다"**(케이스 모달) → **"AI 대조 결과는 참고용입니다"** 로 표준화. ("답변"은 생성형 법률응답 오인 소지)
- **"부당해고로 판정된 공개 사례가 있습니다"**(케이스 모달) → 허용 프레이밍(판정례 사례). ✅ 유지.
- 전반적으로 데모는 "판단하지 않으며 대조 결과·기한만 제공" 고지를 각 화면에 이미 배치 — 우수. 본선에서도 유지.

### (e) 데모가 드러낸 PRODUCT.md 보강 제안 (파일은 안 고침, 제안만)
1. **온프레미스/폐쇄망 + 내부 RAG 포지셔닝**: 데모는 "케이스 데이터는 회사 밖으로 나가지 않음(내부 RAG)"·"폐쇄망 오프라인 업데이트"를 전면에 세움. PRODUCT §1·§6에 **배포 모델(온프레미스/에어갭)** 방향이 없음 → 심사 방어(보안 SaaS 신뢰)·엔터프라이즈 세일즈 포인트로 §1 또는 §6에 명문화 검토.
2. **"대상자" 액터**: 데모는 퇴사자 본인(또는 담당자)이 직접 상신하는 셀프 상신 흐름을 둠. PRODUCT §3-4 오케스트레이터 모델은 인사/담당팀만 언급 → **대상자 셀프 상신 포털** 포함 여부를 §3-4에 반영 검토.
3. **템플릿 구조 명명**: PRODUCT §3-3의 L1/L2/L3 템플릿 3층과 데모의 "레일 템플릿 ↔ 직종·직급 프로파일 ↔ 케이스" 매핑을 용어 정합화 검토.
