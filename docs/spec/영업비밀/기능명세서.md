## TS-01 ★CP — 접근 자산 자동 특정

**목적**: 퇴사자가 접근한 핵심 자산 목록(13건)을 권한·로그에서 자동 특정해 영업비밀 레일의 대조 대상 인벤토리를 만든다.

**액터**: 인사총무 · 담당자 · (시스템)

**사전조건**: 케이스 접수 완료(CM-04) · 데이터 수집(CM-06, read-only 흡수) 완료.

**트리거**: 레일상세(CM-07)에서 영업비밀 탭 진입.

**기본 플로우**:
1. 시스템이 권한대장·접근로그(시뮬 입력)에서 퇴사자 접근 자산을 스캔한다.
2. 자산별로 `TradeSecretAsset` 레코드 생성(name·category·3요소·source). → **13건**.
3. 상단 KPI 3종 표시: 접근 자산 **13** · 보호요건 미충족 **4** · 재서약 미체결 **7**(data-model §2-4).
4. 자산 인벤토리 테이블 렌더(TS-02가 열 구성).

**입력**: 권한대장·접근로그(시뮬). **출력**: `assets[]` + `asset_summary` KPI (`GET /cases/{id}/rails/trade_secret`).

**데이터**: `TradeSecretAsset`(data-model §2). `is_sim=true`(입력=시뮬), 특정 로직=진짜.

**상태전이**: 자산은 상태머신 대상 아님(3요소 파생 상태만). 자산 보완은 TS 검사항목(Item)의 상태머신으로.

**수용 기준(AC)**:
- AC-1: 응답 `asset_summary.asset_count == 13`.
- AC-2: 데모 명시 6건(핵심도면_v7·고객DB_스키마·API_인증키_모음·배포_스크립트_v3·사업계획_2026·인프라_구성도)이 verbatim name·category로 존재.
- AC-3: 나머지 7건은 `is_sim=true` 시뮬 시드로 KPI(미충족 4·재서약 미체결 7)를 정확히 충족.
- AC-4: 각 자산에 특정 근거(`source`) 표기.

**문구경계**: "자동 특정"은 대조용 목록 생성. 자산의 영업비밀 해당 여부를 **판정하지 않음**(그건 TS-05 대조·전문가 몫).

## TS-02 ★CP — 자산별 보호요건 3요소 대조

**목적**: 자산별 비밀표시·재서약·접근제한 3요소를 판례 요구 요소와 대조해 보호요건 충족/미충족을 테이블로 표시한다.

**액터**: 인사총무 · 담당자

**사전조건**: TS-01 자산 인벤토리 존재.

**트리거**: 영업비밀 탭 자산 테이블 렌더.

**기본 플로우**:
1. 자산 6열 테이블 렌더: 자산명 · 분류 · **비밀표시** · **재서약** · **접근제한** · 상태.
2. 3요소 각 셀은 ✓(green) / ✕(red)로 표시(데모: "✓ 표시/✕ 없음", "✓ 완료/✕ 미체결", "✓ 제한/✕ 전원").
3. `protection_status`를 3요소에서 **결정론적 파생**(data-model §2-3)해 상태 배지 렌더:
   - `met` 충족(green) / `re_pledge_needed` 재서약 필요(amber) / `mark_needed` 표시 필요(amber) / `unmet` 요건 미충족(red).
4. 미충족 자산은 보완 진입(재서약 S-02 / 회수 S-07 검사항목 상신-검토).

**입력**: `TradeSecretAsset` 3요소. **출력**: 상태 배지 테이블(TS-01 응답 `assets[]`에 포함).

**데이터**: `AssetProtectionStatus`(data-model §2-2), 파생식 §2-3.

**상태전이(자산 3요소 갱신)**: S-02 재서약 검사항목 `approved` → 연계 자산 `re_pledge=true` → `protection_status` 재파생(예: `re_pledge_needed`→`met`).

**수용 기준(AC)**:
- AC-1: 데모 6행 3요소 조합이 §2-3 파생식으로 데모 배지와 **6행 전부 일치**(핵심도면=unmet, 고객DB=re_pledge_needed, API키=unmet, 배포=met, 사업계획=mark_needed, 인프라=met).
- AC-2: `protection_unmet==4`, `re_pledge_pending==7`.
- AC-3: 재서약(S-02) 확인 완료 시 연계 자산 상태가 재파생되어 배지가 갱신됨.

**문구경계**: 상태 배지는 "요건 미충족/충족" = **요소 대조 결과**. "위법/영업비밀 아님" 단정 금지. "개선방안 제시" 어조 금지(미충족 표시까지, 보완 항목은 기존 템플릿 항목 연결이지 신규 생성형 제안 아님).

## TS-05 ★CP — 비밀관리성 요소 대조 알림

**목적**: "비밀표시 없음 + 전원 접근" 등 판례상 비밀관리성 요구 요소 패턴과 현재 자산 상태를 대조해 미충족을 사례 프레이밍으로 알린다. (AI 판례 대조 = 기술 해자)

**액터**: 인사총무

**사전조건**: TS-01·02 완료(자산 3요소 상태 존재).

**트리거**: 영업비밀 탭 진입 시 자동 대조 / 항목 드로어(CM-09)에서 근거 조회.

**기본 플로우**:
1. 자산 인벤토리 3요소에서 비밀관리성 결여 신호 스캔(data-model §4-2): 비밀표시 없음 · 전원 접근 · 재서약 미체결.
2. 코어 `POST /compare`(rail=trade_secret) 호출 → `CompareResult` 5행 생성.
3. 5행 표시: 절차(신호 요약) · 기준(부경법 §2 인용) · 위험(판례 사례 프레이밍) · 상태(미충족 4건·재서약 미체결 7건) · 출처(링크).
4. 근거 배지 L1(부경법 §2)·L2(비밀관리성 판례)·L3(특허청 매뉴얼) + `boundary_notice` 표시.
5. 판단 필요 시 [전문가 연결] 노출.

**입력**: `CompareInput`(rail=trade_secret). **출력**: `CompareResult`(코어 §6-2, TS 규칙 data-model §4).

**데이터**: `Standard` L1/L2/L3(data-model §5), compare 규칙 §4.

**상태전이**: 없음(조회·대조). 대조 결과는 증적 봉인(CM-12) 대상.

**수용 기준(AC)**:
- AC-1: `rows`가 정확히 5행, kind 순서 `procedure→standard→risk→status→source` 고정.
- AC-2: `risk` 행이 "…비밀관리성이 인정되지 않아 …판단된 사례가 있습니다" 사례 프레이밍(단정 금지).
- AC-3: `boundary_notice`가 코어 §6-3 고정 문구와 **글자 그대로** 일치.
- AC-4: `badges`에 L1·L2·L3 3층 전부 포함(원천 링크).
- AC-5: `status` 행 "미충족 N건" 수치가 자산 인벤토리 KPI와 일치.

**문구경계**: 
- ✅ 허용: "…판정된/판단된 **사례가 있습니다**" · "미충족 N건" · "비밀관리성 요구 요소".
- ❌ 금지: "영업비밀이 아닙니다"(단정) · "패소합니다" · "진단" · "법률 자문".
- LLM은 공개 기준 **인용·링크 위주**. 생성형 법률 판단 배제.
