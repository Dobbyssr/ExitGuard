## 공통 전제 (CM 13기능 공통)

- **주인공(세로절단)**: 김민준 · 백엔드 개발자 · 권고사직(`recommended_resignation`) · D-3. (유비쿼터스 §10 · data-model §0 정본. "김민수"·"김민철" 표기는 stale.)
- **액터 2역할**(유비쿼터스 §8): 인사총무 = 오케스트레이터/허브 = `admin`(데모 "한지수"). 담당자 = `user`(레일 처리·대리 상신). 퇴사자 본인 로그인 없음 — 대상자는 `Case` 데이터.
- **화면 성격**: CM은 **3레일을 담는 셸·공통 인터랙션**이다. 레일 고유 데이터(노무 기한/TS 자산/SEC 회수)는 레일 명세(대표2·3)가 `Item.detail`·레일 엔드포인트로 채우고, CM은 그 결과를 **공통 컴포넌트**(게이트 배너·항목 드로어·상신검토·증적·배지·토스트)로 렌더한다.
- **구현등급**(PRODUCT §6): CM-05(인테이크 대조)는 compare 코어 위 `[진짜구현]`. CM-06(수집 로딩)은 `[시뮬데이터]`(연출). 나머지 CM은 코어 계약 위 `[진짜구현]` 셸·인터랙션. 레일 데이터의 진짜/시뮬 경계는 각 레일 명세가 소유.
- **게이트 시드(데모 정합 · 유비쿼터스 §10)**: 김민준 종합 50% / labor 40% / trade_secret 40% / security 67% / risk 3. **이 %는 저장값이 아니라 항목 상태에서 파생**(data-model §5: labor 2/5·ts 2/5·sec 4/6, 종합 Σ8/Σ16=50%, risk_count=3)되며, 레일별 정합은 시드 항목 상태·blocking으로 조정하는 캘리브레이션 knob이다. CM은 파생된 `Gate` 객체를 렌더할 뿐 수치를 재정의하지 않는다.
- **경계 고지 3종 구분(직역법 · C3 반영)**: ① `CompareResult.boundary_notice`(§6-3 고정문구, 대조 응답 필수) ② `DefenseReport.boundary_notice`(§10, 리포트 필수, ①과 동일 문구) ③ `rail_notice`(레일 상세 응답의 선택적 상시 배너 — 대조가 아닌 레일 고지). 세 필드는 이름이 다르며 CM 화면은 셋을 혼동 없이 표시한다.

## CM-04 ★CP — 퇴사 접수 폼 (3경로 · 사유 4종 · 프로파일 자동적용)

**목적**: 퇴사 발생을 케이스로 등록한다. 접수 경로 3종(그룹웨어 자동감지 / 해고·권고 수동 / 사직서 수동)과 사유 4종을 받아 `Case`를 생성하고, 직무·직급 프로파일로 3레일 검사항목을 자동 배정한다("빈 종이 빌더 금지").

**화면동작**:
- 접수 경로 3종을 선택한다(`intake_route`, 유비쿼터스 §9):
  - `groupware`(자동감지): 그룹웨어 전자결재에서 사직서·사유·희망 퇴직일을 **read-only 추출**(원본 미변경)하고 프로파일 템플릿을 자동 매칭. 화면 안내: "AI가 각 문서에서 사유·희망 퇴직일을 추출하고 템플릿을 자동 매칭했습니다 (read-only)".
  - `dismissal`(수동 · 회사 결정): 해고·권고사직은 직원 상신이 없어 인사담당자가 직접 등록. 결정 근거·통보 방식이 증적에 반영됨을 콜아웃.
  - `resignation`(수동 · 사직서 접수): 서면·메일 사직서를 접수·스캔 첨부해 등록. 원본은 증적 패키지에 봉인됨을 콜아웃.
- 회사 정보·사유 유형 입력: 대상자 이름·직무·직급·직책, 사유 유형 칩 4종(`voluntary` 자발 / `recommended_resignation` 권고사직 / `dismissal` 해고 / `contract_expiry` 계약만료), 회사 사유 텍스트(`reason_text` — CM-05 인테이크 대조·compare 엔진의 입력원).
- 선택 요약 바: "사유 {유형} · 직무 {job} · 직급 {rank}".
- 접수 확정 시 `POST /cases`(api-spec §2-1) → `profile_id`의 `rail_map`으로 각 RailTemplate의 TemplateItem을 `Item`으로 복제(상태 `pending`, 유형 규칙상 일부 `not_applicable`). 예: 권고사직이면 노무 L-09(해고예고 30일)는 `not_applicable`.
- 접수 성공 → CM-06 수집 로딩 연출로 전이 + `meta.toast`(CM-32).

**상태전이**: `Case.status` 초기 `in_progress`(data-model §3-1-1 상태머신 진입점). 항목은 전부 `pending`(대상 아님은 `not_applicable`, data-model §4). CM-04 자체는 케이스·항목을 **생성**하며 이후 전이는 상신-검토(CM-10)·승인(CM-08)이 소유.

**수용 기준(AC)**:
- AC-1: 3경로(`groupware`/`dismissal`/`resignation`) 중 하나로 접수 시 `Case.intake_route`가 그 값으로 저장된다.
- AC-2: 사유 4종이 `ExitReason` enum(data-model §2-2)에 1:1 매핑되고, 임의 제5의 사유를 만들지 않는다.
- AC-3: `profile_id` 적용 시 rail_map의 3레일 TemplateItem이 `Item`으로 복제되어 3레일에 분산 생성된다(공통 `Case.apply_profile()`, data-model §8).
- AC-4: 권고사직 케이스에서 해고 전용 항목(L-09 해고예고)이 `not_applicable`로 생성된다(유형 규칙, data-model §4).
- AC-5: 대상자 이름 누락 시 `400 VALIDATION_ERROR`(`fields.subject_name=required`, api-spec §1-3).
- AC-6: `groupware` 경로는 그룹웨어 원본을 변경하지 않는다(read-only 추출만).

**문구경계**: "자동감지·자동 매칭"은 **read-only 추출·템플릿 매칭**을 뜻하며 법적 성격 판정이 아니다. 사유 유형 선택은 회사 입력 사실의 기록이지 "이 퇴사는 해고다/부당하다" 판단이 아니다. "진단·판정" 라벨 금지.

## CM-05 ★CP — 인테이크 대조 패널 (5행 표시)

**목적**: 접수된 회사 사유 텍스트(`reason_text`)를 즉시 공개 기준과 대조해, 절차 리스크 신호가 있으면 5행 대조 결과를 접수 흐름 안에서 미리 보여준다(레일 진입 전 조기 알림). 공용 compare 엔진(data-model §6·§6-4) 위에 얹는다.

**화면동작**:
- 접수 직후 `POST /cases/{id}/intake-compare`(api-spec §2-3, 내부적으로 `POST /compare` 호출, `case_facts` 자동 채움) 결과를 패널로 렌더.
- **정확히 5행**(`CompareRowKind` 고정 순서, data-model §6-2): `procedure`(절차) → `standard`(기준 대조) → `risk`(위험) → `status`(상태 대조) → `source`(출처 보기).
  - 신호 감지 예(해고성 사유): 절차="회사 사유 텍스트에서 '구두 통보'·'서면 없음' 감지 → 서면통지 요건 미이행 가능성" / 기준 대조=근기법 §27 원문 인용 / 위험="서면통지 누락 시 사유를 불문하고 부당해고로 판정된 **공개 사례가 있습니다**" / 상태="미충족 N건" / 출처="원문 링크".
  - 신호 미감지(김민준 권고사직 기본 서사): 절차="퇴사 사유에서 절차 리스크 신호 미감지 → 법정 기한 항목만 관제 대상으로 분류" / 기준 대조=근기법 §36(금품청산 14일) 인용 / 상태=기한 D-day. (§36은 status D-day로만, risk 행 생략 가능 — 레일 규칙 소유, 노무 LB-04 AC-6.)
- `badges`(L1/L2/L3, CM-15) + `boundary_notice`(§6-3 고정문구) + 판단 분기 시 [전문가 연결](`expert_referral`, api-spec §1-7) 노출.
- 다중 토픽 신호 시 `data`=[CompareResult] 배열로 여러 패널 표시.

**상태전이**: 없음(조회·대조). 단, 확정된 `CompareResult`는 **자동 증적 봉인**된다(`event_type=compare_recorded`, data-model §3-4-1) → 방어 리포트(CM-13)가 재계산 없이 인용.

**수용 기준(AC)**:
- AC-1: `rows`가 정확히 5행이고 kind 순서가 `procedure→standard→risk→status→source`로 고정(data-model §6-2). 4행·6행·순서 뒤바뀜은 실패.
- AC-2: compare 파이프라인이 **① LLM 신호추출 → ② 결정론 조립**(data-model §6-4)을 거친다. `risk` 행 문구는 결정론 템플릿에 값 치환이며 LLM 자유서술이 아니다.
- AC-3: `boundary_notice`가 §6-3 고정 문구와 **글자 그대로** 일치.
- AC-4: LLM 실패·환각·유효신호 0건 시 **결정론 fallback**으로 5행 shape·`boundary_notice`가 유지된다(`risk` 생략 가능, api-spec §2-3). compare가 깨져 화면이 빈 상태가 되지 않는다.
- AC-5: 대조 응답에 `badges`(L1/L2/L3)가 누락 없이 포함(CM-15 · "GPT 래퍼 아님").

**문구경계**: `risk` 행은 **단정 금지** — "…로 판정된 사례가 있습니다" 프레이밍만("부당해고입니다"/"위법입니다"/"패소합니다" 금지). `procedure`는 "신호 감지·가능성" 어조, `status`는 "미충족 N건·D-day" 대조 어조("개선방안 제시" 금지 — 노무관리진단 정의 회피). LLM 산출은 공개 기준 인용·링크만.

## CM-06 ★CP — 수집 로딩 연출

**목적**: 접수 직후 데이터 수집·기준 대조가 진행 중임을 4단계 순차 연출로 보여준다(체감 신뢰·"백그라운드에서 무엇을 하는지"의 투명성).

**화면동작** `[시뮬데이터]`:
- 접수 확정 후 4단계를 순차 표시(각 단계 완료 시 ✓, 진행 중은 순번, 데모 순차 타이밍 ~650ms):
  1. "GroupWare PRO — 전자결재·인사·조직 조회"
  2. "오피스키퍼 — 반출 로그·자산 조회"
  3. "권한 테이블·서약서 문서함 조회"
  4. "내부 RAG — L1 법령·L2 판례·L3 가이드 대조"
- 4단계 완료 → 레일상세 셸(CM-07)로 전이.
- 데이터 소스: `POST /cases` 접수 응답(생성된 items 요약·초기 gate) + 프론트 연출. **별도 수집 엔드포인트 없음**(시뮬 연출, api-spec §4 CM-06=프론트 연출).

**상태전이**: 없음(연출). 케이스는 이미 CM-04에서 `in_progress`.

**수용 기준(AC)**:
- AC-1: 4단계가 순서대로 표시되고 마지막 단계 완료 시 레일상세(CM-07)로 자동 전이.
- AC-2: 연출은 시뮬이며, 실제 데이터는 접수 응답(items·gate)에서 온다(수집 연출이 케이스 데이터를 만들지 않음).
- AC-3: 4단계 라벨이 데모 문구와 일치(외부 연동원 3종 + 내부 RAG 대조).

**문구경계**: 4단계 라벨은 **조회·대조 과정의 표현**이며 판정 결과가 아니다. "RAG 대조"는 공개 기준 인용 준비를 뜻하며 "AI가 위법을 진단 중" 류 표현 금지.

## CM-07 ★CP — 레일상세 셸 (3레일 탭 · 게이트 배너)

**목적**: 한 케이스의 3레일(노무/영업비밀/보안)을 탭으로 담고, 상단에 통합 게이트 배너를 고정하는 컨테이너 화면. 레일 명세(대표2·3)의 상세 뷰가 이 셸의 탭 안에 렌더된다.

**화면동작**:
- `GET /cases/{id}`(api-spec §2-1) 응답으로 `case`·`gate`·`rails{labor,trade_secret,security}`·`items`를 로드.
- 상단: 통합 게이트 배너(CM-08) — 종합 완료율·미해소 리스크·방어 가능 상태.
- 탭 3개: 노무 / 영업비밀 / 보안. 각 탭 레이블에 레일별 완료율(`gate.rail_completion[rail]`)·미충족 배지. 탭 본문은 각 레일 상세(`GET /cases/{id}/rails/{rail}`, 대표2·3 소유) — CM은 탭 프레임·전환만 소유.
- 레일 탭 진입 시 해당 레일의 `rail_notice`(선택적 상시 배너)가 있으면 표시(예: 보안 "확인 기록이지 회수 실행 아님", 유비쿼터스 §9-P-j).
- 항목 클릭 → 항목 드로어(CM-09).

**상태전이**: 없음(조회 셸). 게이트·항목 상태는 파생·하위 인터랙션이 갱신.

**수용 기준(AC)**:
- AC-1: 3레일 탭이 `labor`/`trade_secret`/`security` 순으로 렌더되고, 각 탭이 해당 `rails.<rail>`·완료율을 표시.
- AC-2: 게이트 배너(CM-08)가 상단에 고정되어 탭 전환과 무관하게 종합 상태를 유지.
- AC-3: 레일 코드 식별자는 `trade_secret`(축약형 `secret` 아님, 유비쿼터스 §2)로 통일.
- AC-4: `rail_notice`(레일 상시 배너)와 `boundary_notice`(대조 고정문구)를 혼동 없이 구분 표시(C3 · 유비쿼터스 §9-P-j).

**문구경계**: 게이트·탭 레이블의 완료율은 **항목 충족 상태의 기계적 집계**이지 적법성 표시가 아니다. "방어 가능"의 경계 고지(CM-08)를 셸 레벨에서 승계.

## CM-08 ★CP — 게이트 판정 표시 (방어가능 / 미해소 N건)

**목적**: 케이스의 3레일 항목 상태에서 결정론적으로 파생한 통합 게이트(data-model §5)를 종합 표시하고, "방어 가능 상태" 여부와 미해소 리스크 수를 알린다. 승인(퇴사 확정)의 관문이다.

**화면동작**:
- `GET /cases/{id}/gate`(api-spec §2-2) 또는 케이스 상세 응답의 `gate`를 렌더: `overall_completion`·`rail_completion{labor,trade_secret,security}`·`risk_count`·`defensible`.
- 표시 분기(data-model §5):
  - `defensible == true`(risk_count 0) → "방어 가능 상태" + [퇴사 승인] 진입 가능.
  - `risk_count > 0` → "관제 중 · 미해소 리스크 N건"(강조).
- 김민준 데모: 종합 50% / labor 40% / ts 40% / sec 67% / risk 3 → "미해소 리스크 3건". (시드 캘리브레이션, data-model §5 — risk 3건 전부 노무 blocking 미충족 L-04·L-06·L-08.)
- **승인 액션(T3 이음새)**: [퇴사 승인]은 `POST /cases/{id}/approve`(api-spec §2-1) 호출. 서버가 **승인 조건**(data-model §3-1-1 `approvable = defensible && submitted 항목 0건`)을 강제 — 조건 미충족 시 `409 INVALID_TRANSITION`(미충족 사유 risk_count·submitted 수 포함) 반환하고 화면은 그 사유를 안내. **게이트 우회 강제 승인 없음.**
- 승인 성공 → `Case.status → completed` + 자동 증적 봉인(`case_approved`, data-model §3-4-1) + `meta.toast`="✓ 방어 가능 상태로 승인되었습니다".

**상태전이**: 게이트 자체는 mutation 없음(순수 파생). 승인 확정만 `Case.status: (review_waiting|in_progress) → completed`(data-model §3-1-1). 승인은 `approvable==true`에서만.

**수용 기준(AC)**:
- AC-1: 게이트 4필드가 data-model §5 계산식(`risk_count = 미충족 blocking 항목 수`, `defensible = risk_count==0`)대로 파생되어 표시.
- AC-2: `risk_count > 0`일 때 [퇴사 승인]을 시도하면 `409 INVALID_TRANSITION`으로 거부되고, 응답의 미충족 사유가 화면에 안내된다(강제 승인 불가).
- AC-3: `submitted`(미검토 상신) 항목이 1건이라도 있으면 `defensible`이 true여도 승인이 거부된다(`approvable` 두 조건 동시 강제, data-model §3-1-1).
- AC-4: 승인 성공 시 `Case.status`가 `completed`가 되고 `case_approved` 증적이 자동 봉인된다(게이트 스냅샷·승인자·시각).
- AC-5: 게이트 %가 저장 하드코딩이 아니라 항목 상태에서 파생된다(상신→확인으로 항목 충족 시 %·risk_count가 재계산).

**문구경계**: "방어 가능 상태"는 *공개 기준 대비 항목 충족 + 검토 완료* 상태이며 **적법·승소를 보증하지 않는다**(data-model §5·§3-1-1 경계). 이 고지를 배너에 명시. "위법 없음/합법 확정" 류 단정 금지. "미해소 리스크 N건"은 미충족 blocking 항목 수의 대조 표현이지 위법 건수 판정이 아니다.

## CM-09 ★CP — 검사항목 드로어 (근거 · 상신 진입)

**목적**: 한 검사항목의 상세(상태·근거·확인요건·상신 이력)를 드로어로 열고, 담당자의 상신(CM-10) 진입점을 제공한다.

**화면동작**:
- 항목 클릭 → `GET /items/{id}`(api-spec §2-4) 응답으로 `Item` + `badges` + `approvals`(이력) + `basis`(확인요건·근거) 로드.
- 표시: 항목명·코드(L-/S-/C-)·유형 배지(법정/내규/권고, `ItemKind`)·현재 상태(`ItemStatus`)·진행 요약(`sub`)·기한(`deadline`, 있으면 D-day)·근거 배지(CM-15)·기준 근거 문구.
- 상태별 진입:
  - `pending`/`rejected` → [상신] 버튼 → 상신 시트(CM-10, `POST /items/{id}/submit`).
  - `submitted` → 관리자에게 [검토](확인/반려) 노출(CM-10, `POST /items/{id}/review`).
  - `approved`/`not_applicable` → 조회 전용(이력·근거 표시).
- 레일 고유 상세 필드(`Item.detail`: 노무 기한요소/TS 자산3요소/SEC 회수대상)는 레일 명세가 채운 스키마를 드로어가 렌더(CM은 슬롯만 노출, 내용은 대표2·3 소유).

**상태전이**: 드로어 자체는 조회. 전이는 드로어에서 진입한 CM-10 상신-검토가 트리거(`Item.status` 상태머신, data-model §4).

**수용 기준(AC)**:
- AC-1: 드로어가 `Item` + `badges` + `approvals` + `basis`를 표시(api-spec §2-4).
- AC-2: 항목 상태에 따라 진입 버튼이 분기([상신] / [검토] / 조회전용)되고, 상태머신상 불가능한 액션은 노출되지 않는다.
- AC-3: 근거 배지(L1/L2/L3)와 기준 근거 문구가 표시된다(근거 없는 판정 표시 금지).
- AC-4: 레일 고유 `detail` 필드는 레일 명세가 정의한 스키마대로 표시되며 CM이 임의 필드를 추가하지 않는다.

**문구경계**: 드로어의 "미충족 리스크" 표현은 상태 대조 결과이지 위법 판정이 아니다(api-spec §1-7). 근거는 **인용·출처**로 표시하고 "이 항목은 위법" 류 단정 금지.

## CM-10 ★CP — 상신-검토 인터랙션 (상신 → 검토 → 확인/반려)

**목적**: 담당자가 항목 처리 사실을 상신하고 관리자가 검토(확인/반려)하는 이원 승인 워크플로우. 처리 사실을 증적으로 자동 봉인하는 핵심 인터랙션이다.

**화면동작**:
- **상신**(`user`, `POST /items/{id}/submit`, api-spec §2-4): 상신 시트에서 메모(`memo`)·첨부(`attachments[{name,size}]`)·전자서명(`signed`) 입력 → `Approval` 생성, `Item.status: pending|rejected → submitted`. 다건 일괄 상신 지원(데모 "N건 일괄 상신").
- **검토**(`admin`, `POST /items/{id}/review`, api-spec §2-4): 검토 시트에서 출처 템플릿·검사항목·대상자 메모·상신 자료(열람)·전자서명·기준 대조 근거를 확인 후:
  - [확인 완료] → `decision=confirmed` → `Item.status: submitted → approved`.
  - [반려 · 재상신 요청] → `decision=rejected`(+memo) → `Item.status: submitted → rejected`. 재상신은 `POST /items/{id}/submit` 재호출.
- 각 액션 후 `meta.toast`(CM-32) + 게이트(CM-08)·상태 대조 재파생.
- **증적 자동봉인(T2 이음새)**: 각 처리는 서버가 **동일 트랜잭션에서 자동 봉인**(data-model §3-4-1) — 상신=`item_submitted`, 확인=`item_confirmed`, 반려=`item_rejected`. 별도 봉인 호출 불요. payload에 항목·상신 memo·첨부메타·전자서명·검토자·basis_note 스냅샷.

**상태전이**(data-model §4 · 재정의 아님, 인용):
```
pending ──상신──► submitted ──확인(confirmed)──► approved
rejected ──재상신──► submitted ──반려(rejected)──► rejected
```
- 불변식: `approved`/`not_applicable`만 게이트 "미충족 아님"으로 집계(data-model §5). 이미 `approved`인 항목 재상신은 `409 INVALID_TRANSITION`.

**수용 기준(AC)**:
- AC-1: 상신 시 `Approval`이 생성되고 `Item.status`가 `submitted`로 전이(`pending`/`rejected`에서만 가능).
- AC-2: 확인 시 `approved`, 반려 시 `rejected`로 전이하고, 반려 후 재상신이 다시 `submitted`로 돌아온다.
- AC-3: 상신·확인·반려 각각에 대해 대응 `event_type`(`item_submitted`/`item_confirmed`/`item_rejected`) 증적이 **자동 봉인**된다(data-model §3-4-1). 봉인 누락 0.
- AC-4: 자동 봉인은 처리 mutation과 동일 트랜잭션에서 이뤄져 이력 누락이 없다(seq 단조증가·prev_hash 체인).
- AC-5: `submitted`가 아닌 항목에 검토 시도 시 `409 INVALID_TRANSITION`, `user`가 review 시도 시 `403 FORBIDDEN`.
- AC-6: 상신 시트의 전자서명(`signed=true`)·첨부 메타가 봉인 payload에 포함된다.

**문구경계**: 검토의 "기준 대조" 문구는 공개 기준 인용이며 "이 처리가 적법하다는 확인"이 아니다. 반려 사유는 **보완 필요 항목의 지정**이지 생성형 개선방안·법률 지도가 아니다(노무관리진단 정의 회피). "확인 완료"는 처리 사실 확인이지 위법성 면제가 아니다.

## CM-12 ★CP — 증적 봉인 뷰 (아카이브)

**목적**: 케이스의 처리 이력이 자동 봉인된 증적(SHA-256 해시 체인)을 시간순으로 보여주고, 봉인 상태·무결성을 표시한다("분쟁·소송 시 export"의 원천).

**화면동작**:
- `GET /cases/{id}/evidence`(api-spec §2-5) 응답으로 `Evidence[]`(seq 순, `event_type`·`origin` 포함) + `meta.seal_status`(`sealed`/`accruing`) + 항목수·봉인일시·`head_hash`(체인 헤드) 로드.
- 아카이브 목록: 케이스·유형·봉인 일시·항목 수·상태·[export]. 각 증적 행: seq·처리 일시(`occurred_at`)·수행자(`actor`)·처리 내용(`action`)·`event_type`·`origin`(auto/manual)·`integrity_hash`.
- **자동 봉인이 기본**(data-model §3-4-1): 상신/확인/반려/대조확정/회수확인/케이스승인 6종 이벤트가 자동 append. 수동 보충 봉인은 `POST /cases/{id}/evidence`(`origin=manual`, `admin`)로만.
- 무결성 표시: `integrity_hash = SHA-256(canonical_json(payload) + prev_hash)`, `prev_hash` 체인으로 순서 위변조 탐지 가능함을 표기(변경 불가·append-only).

**상태전이**: 증적은 append-only(생성 후 변경 불가). Case 단위 `seal_status`는 파생(`sealed`/`accruing`). CM-12는 조회 + 수동 보충 봉인 진입만.

**수용 기준(AC)**:
- AC-1: 증적이 `seq` 순으로 표시되고 각 행에 `event_type`·`origin`·`integrity_hash`가 포함된다.
- AC-2: `head_hash`(체인 헤드)와 `seal_status`가 표시되어 무결성 앵커·봉인 상태를 확인할 수 있다.
- AC-3: 증적은 변경·삭제 UI가 없다(append-only 계약, data-model §3-4).
- AC-4: 수동 보충 봉인(`POST /evidence`)은 `admin`만 가능하고 자동 봉인과 동일 체인에 append된다(seq 연속).
- AC-5: 자동 봉인 6종(`item_submitted`/`item_confirmed`/`item_rejected`/`compare_recorded`/`recovery_confirmed`/`case_approved`)이 해당 이벤트 발생 시 목록에 나타난다.

**문구경계**: 증적은 **처리 사실의 봉인 기록**이며 법적 판단이 아니다(data-model §3-4-1 경계). payload 서술 필드는 단정·진단 표현 금지("금품청산 정산 확인완료" 같은 사실 기록만, "적법 확인" 금지).

## CM-13 ★CP — 방어 리포트 Export

**목적**: 봉인 증적·게이트·compare 스냅샷을 하나의 방어 리포트(`DefenseReport`, data-model §10)로 조립해 JSON/PDF로 export한다("소송에서 쓸 봉인된 기록").

**화면동작**:
- [리포트 생성]/[export] → `GET /cases/{id}/evidence/export`(api-spec §2-5, B1). `?format=json`(기본) 또는 `?format=pdf`(`application/pdf`).
- 리포트 구성(`DefenseReport`, data-model §10): `case` 요약 · `kpi`(게이트 파생 — 봉인된 `case_approved` 스냅샷 우선) · `rails`(3레일 완료율·미충족·배지) · `compare_findings`(봉인된 `compare_recorded` 스냅샷 인용 — 재계산·환각 없음) · `evidence_chain`(체인 요약·`head_hash`) · **`boundary_notice`(필수)**.
- 데모 표시: "봉인 증적 기반 방어 가능 상태" · 방어 가능률 · 봉인 증적 건수 · 기준 DB 버전 · "export 형식: PDF · 원본".
- 리포트 표지·문구에 "방어 가능 상태 ≠ 적법·승소 보증" 경계 고지 명시.

**상태전이**: 없음(파생 뷰 조립·export). `DefenseReport`는 **저장 엔티티 아님**(data-model §10) — 봉인 증적·게이트·compare 스냅샷에서 조립.

**수용 기준(AC)**:
- AC-1: 응답에 `boundary_notice`(§6-3 고정 문구)가 **필수 포함**된다. 누락 시 리포트 무효(api-spec §2-5, data-model §10).
- AC-2: `compare_findings`가 실시간 재계산이 아니라 봉인된 `compare_recorded` 스냅샷을 인용한다(재현성·환각 배제).
- AC-3: `evidence_chain.head_hash`가 포함되어 봉인 이력 위변조를 재검증할 수 있다(SHA-256 체인 앵커).
- AC-4: `kpi.defensible`이 표시되되 "적법·승소 보증"으로 읽히지 않게 경계 고지가 승계된다(data-model §10·§3-1-1).
- AC-5: `format=json`·`format=pdf` 두 형식이 동일 `DefenseReport` 뷰를 반환/렌더한다.

**문구경계**: 리포트는 외부(법원·상대방) 제출 가능 문서 → `boundary_notice` **고정 문구를 글자 그대로 승계**(직역법 §4 최고강도). "방어 가능"을 "합법·승소"로 표기 금지. KPI·완료율은 항목 충족 상태의 집계이지 판정이 아니다.

## CM-15 ★CP — 근거 배지 (L1/L2/L3)

**목적**: 모든 대조·항목·리포트 응답에 근거 배지(L1 법령 / L2 판례·판정례 / L3 정부 가이드)를 붙여 판정 근거를 출처로 노출한다("GPT 래퍼 아님" 증명 — 근거 없는 판정 0).

**화면동작**:
- `badges` 규약(api-spec §1-6): `[{ tier, title, url, version }]`. `tier ∈ L1|L2|L3`(`StandardTier`, data-model §2-5).
- 렌더: tier로 배지 색·라벨(L1/L2/L3), 클릭 시 `url`(원문 링크), `version` 표기(예: `v2026.07`).
- 노출 지점: 인테이크 대조(CM-05) · 항목 드로어(CM-09) · 레일 대조(compare) · 리포트(CM-13) · 기준 DB(CM-16) — 근거가 붙는 모든 응답.
- 배지 원천은 `Standard`(data-model §3-5). CM은 배지 컴포넌트·표시 규약만 소유(레일별 근거 시드는 대표2·3).

**상태전이**: 없음(표시 규약).

**수용 기준(AC)**:
- AC-1: 대조/항목/리포트 응답에 `badges`가 누락 없이 포함되고, 각 배지가 `{tier,title,url,version}` 4필드를 갖는다.
- AC-2: `tier`가 `L1|L2|L3` enum 값과 일치하고, 프론트가 tier로 색·라벨을 렌더한다.
- AC-3: 배지의 `url`이 원문 링크로 연결된다(출처 추적 가능).
- AC-4: 근거가 확정되지 않은 지점은 `[시드확인필요]`로 두되 확정된 tier만으로 배지가 성립한다(창작 금지 — 없는 근거를 지어내지 않음).

**문구경계**: 배지는 **출처 표기** 전용이며 배지 자체가 판단을 서술하지 않는다. "진단 결과·판정 근거" 류 라벨 대신 "적용 법령·참조 판례·정부 가이드" 표현.

## CM-16 ★CP — 기준 DB 화면 (3층 기준 스택)

**목적**: 레일 판정의 근거가 되는 3층 기준 스택(L1 법령 / L2 판례·판정례 / L3 정부 가이드·표준서식)과 버전·갱신 상태를 조회·관리하는 화면. 배지(CM-15)의 원천 데이터.

**화면동작**:
- `GET /standards`(api-spec §2-6). `?rail=&tier=L1|L2|L3&q=&page=&size=` 필터. 응답 `data`=[`Standard`: tier·title·article·source_url·version·updated_at] + `meta.pagination`.
- 요약 카드(데모): "최신 반영 2026-07-01" · L1 법령(국가법령정보센터) [최신] / L2 판례·판정례 DB [갱신 예정] / L3 정부 가이드·표준서식 [최신].
- 상세: 조문/요지(`article`)·인용 원문(`body`)·원문 링크(`source_url`)·버전(`version`)·최신 반영일(`updated_at`). 버전 카드·갱신 이력은 이 데이터 파생.
- 레일 필터로 노무/영업비밀/보안 근거를 분리 조회.

**상태전이**: 없음(조회). 기준 시드 관리는 레일 명세(대표2·3)가 레코드를 채움.

**수용 기준(AC)**:
- AC-1: `GET /standards`가 L1/L2/L3 3층·rail·q 필터·페이지네이션을 지원한다.
- AC-2: 각 기준 레코드가 `title`·`source_url`·`version`·`updated_at`를 갖고 원문 링크로 연결된다.
- AC-3: 기준 DB의 데이터가 CM-15 배지의 원천과 동일하다(배지 title/url/version이 기준 DB 레코드에서 파생).
- AC-4: 버전·갱신 상태(최신/갱신 예정)가 표시된다(근거의 시의성 = 신뢰).

**문구경계**: 기준 DB는 **공개 기준의 인용·버전 관리**이며 판정 로직이 아니다. "이 법령으로 위법 판정" 류 표현 금지 — "적용 근거·버전"으로 표기.

## CM-22 ★CP — 레일 템플릿 화면 (프로파일 · Fork & Customize)

**목적**: 직종·직급 프로파일이 레일별로 어떤 템플릿(검사항목 묶음)을 적용하는지 보여주고 관리한다. 접수(CM-04) 시 이 매핑으로 항목이 자동 배정된다("빈 종이 빌더 금지 · Fork & Customize").

**화면동작**:
- 2계층 구조(data-model §3-6·§3-7): `Profile`(직종·직급 → 레일별 `RailTemplate` 매핑, `rail_map`) → `RailTemplate`/`TemplateItem`(그 템플릿의 항목).
- 프로파일 예(데모): "개발직 · 시니어 이상" → labor:"해고 대응 템플릿" / trade_secret:"개발·연구직 강화 템플릿" / security:"개발직 강화 템플릿".
- 템플릿 항목 예(데모): 기본 노무 템플릿 = L-01 사직 합의서 서면 확인·L-02 연차 미사용 수당 정산·L-04 금품청산(14일)·L-06 이직확인서 발급·L-08 4대보험 상실 신고·L-09 해고예고(30일), 각 유형 배지(법정/내규/권고, `ItemKind`).
- 관리: 레일별 템플릿 선택/해제, 템플릿 항목 추가/제외(우리 제공 기본 `is_base=true`는 L1 큐레이션=해자 / 회사 커스텀 구분). 저장 시 `meta.toast`.
- **접수 이음새**: 케이스가 `profile_id`를 받으면 `rail_map`의 각 RailTemplate의 TemplateItem을 `Item`으로 복제(`Case.apply_profile()`, CM-04). CM-22는 그 원천 라이브러리 화면.
- 레일별 상세 항목 필드(`TemplateItem.detail_schema`)는 대표2·3이 채움 — CM은 템플릿·항목·매핑 구조만 소유.

**상태전이**: 없음(템플릿 CRUD). 템플릿 자체는 케이스 상태와 무관(항목 라이브러리).

**수용 기준(AC)**:
- AC-1: 프로파일이 레일 3종 각각에 `RailTemplate`을 매핑하고(`rail_map`), 화면에 매핑 상태가 표시된다.
- AC-2: 템플릿 항목이 `code`·`name`·`kind`·`blocking`을 갖고 렌더된다(data-model §3-6).
- AC-3: 접수(CM-04) 시 선택된 프로파일의 rail_map대로 3레일 `Item`이 복제 생성된다(CM-22 ↔ CM-04 정합).
- AC-4: 기본 제공 템플릿(`is_base`)과 회사 커스텀이 구분 표시된다.

**문구경계**: 템플릿·항목은 **검사 체크리스트의 정의**이며 판정 기준의 창작이 아니다. 항목 유형(법정/내규/권고)은 필수성 구분이지 "위반 여부" 표시가 아니다.

## CM-32 ★CP — 토스트

**목적**: 서버 mutation(접수·상신·검토·승인·저장 등) 후 결과를 짧은 토스트로 피드백한다.

**화면동작**:
- 규약(api-spec §2-7): 서버 mutation 응답의 `meta.toast`(선택) 문자열을 하단 중앙 토스트로 표시. 별도 엔드포인트 없음.
- 프리픽스 관용(데모): 성공 `✓`(예: "✓ 김민준 케이스가 등록되었습니다" · "✓ 방어 가능 상태로 승인되었습니다") / 되돌림 `↩`(예: "↩ 레일 템플릿을 해제했습니다") / 주의 `⚠️`(예: "⚠️ 항목 이름을 입력하세요").
- 자동 소멸(짧은 표시 시간).

**상태전이**: 없음(피드백 UI).

**수용 기준(AC)**:
- AC-1: mutation 응답에 `meta.toast`가 있으면 토스트가 표시되고 자동 소멸한다.
- AC-2: 성공/되돌림/주의가 프리픽스로 구분된다.
- AC-3: 토스트는 서버 응답의 `meta.toast` 값을 그대로 표시하며 클라이언트가 판정 문구를 창작하지 않는다.

**문구경계**: 토스트는 **처리 결과 피드백**이다. "✓ 방어 가능 상태로 승인되었습니다"는 항목 충족·검토 완료 상태의 확인이지 적법 확정이 아니다. "위법 없음 확인" 류 단정 금지.
